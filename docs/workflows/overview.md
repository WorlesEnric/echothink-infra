# EchoThink 工作流体系总览

## 1. EchoThink 的工作流设计哲学

EchoThink 的核心设计哲学可以用一句话概括：AI 不是工具，而是结构化的员工。在传统的软件开发流程中，AI 通常被视为一种辅助手段——开发者在需要时调用它，获取一段代码建议或一篇文案草稿，然后将结果复制粘贴到自己的工作环境中。这种交互模式本质上是单向的、碎片化的，AI 既没有持久的身份，也没有对项目全局状态的感知能力。EchoThink 彻底颠覆了这种范式。在 EchoThink 的体系中，每一个 AI Agent 都拥有由 Authentik 颁发的独立身份和权限凭证，拥有在 Supabase 中可被追溯的工作记录，拥有通过 Dify 和 n8n 定义的标准化工作流程，拥有由 Graphiti 和 Outline 构建的长期记忆与知识储备，拥有在 Hatchet 中被调度和监控的执行环境，以及由 Langfuse 提供的全链路可观测性。换言之，一个 AI Agent 在 EchoThink 中的存在方式，与一个人类团队成员几乎没有本质差异——它有名字、有职责、有权限边界、有工作产出、有绩效记录。

这种设计哲学的深层动机来源于游戏开发团队的实际需求。游戏开发是一项极其复杂的创造性工程，涉及叙事设计、关卡策划、美术概念、程序实现、质量保障等多个高度专业化的领域。一个小型独立团队往往只有十几个人，却需要覆盖上述所有领域的工作。EchoThink 的愿景是让 AI Agent 真正填补这些能力缺口——不是作为偶尔被调用的聊天机器人，而是作为全天候在岗的专职员工，持续阅读项目文档、生成设计资产、编写和审查代码、执行测试用例、更新知识图谱。人类团队成员的角色从"亲自完成所有工作"转变为"指导、审核和协调一支由人类与 AI 混合组成的团队"。这种协作不是简单的自动化，而是真正意义上的共同创造。整个工作流体系的设计都围绕这一哲学展开：如何让人类和 AI 在同一个项目空间中高效协作，如何确保 AI 的行为可控、可审计、可追溯，如何在赋予 AI 足够自主权的同时保留人类的最终决策权。

## 2. Claw Cluster — 智能核心的定位与愿景

Claw Cluster 是 EchoThink 体系中正在开发的一个核心模块，它的定位是整个系统的智能中枢——一个由多个专业化 AI Agent 组成的独立员工集群。如果说 EchoThink 的基础设施层提供了"办公楼"和"办公设备"，那么 Claw Cluster 就是在这栋楼里工作的"AI 员工团队"。这个集群中的每一个 Agent 都有明确的专业分工：概念美术师负责根据设计文档生成视觉概念；叙事设计师负责构建和维护游戏的世界观、角色关系和剧情线索；关卡设计师负责根据玩法规则和叙事需求产出关卡布局与难度曲线；QA 测试员负责运行自动化测试套件并分析测试结果中的异常模式；Code Reviewer 负责审查 GitLab 上的 Merge Request，检查代码质量、安全隐患和架构一致性。这些 Agent 并非各自孤立运作，而是通过 EchoThink 的基础设施紧密协作，形成一个有机的整体。

Claw Cluster 的核心特征在于它对整个 EchoThink 服务栈拥有全面的读写访问能力。具体而言，Claw Cluster 中的 Agent 能够从 Outline 知识库中读取最新的游戏设计文档、技术规范和会议纪要，确保自己的工作始终基于团队最新的决策和共识。当需要执行复杂的多步骤任务时，Agent 会通过 Dify 工作流引擎调用预定义的工作流程，例如"根据角色描述生成概念美术"或"根据关卡需求文档生成初始布局方案"。生成的文件——无论是图片、文档还是代码文件——都会被存储到 MinIO 对象存储中，通过统一的 S3 兼容接口进行管理。当 Agent 完成代码编写或修改后，它会直接将代码提交到 GitLab 仓库，创建 Merge Request 并附上详细的变更说明，等待人类开发者审查。所有任务的调度、执行和重试逻辑都由 Hatchet 任务引擎管理，确保即使某个 Agent 在执行过程中遇到故障，任务也能被自动恢复并继续执行。在知识管理方面，Agent 会将工作过程中获取的新知识——新的设计决策、技术发现、项目关系变更——写入 Graphiti 时序知识图谱，使整个团队的知识储备持续增长和更新。所有 LLM 调用都通过 LiteLLM 网关统一路由，这意味着团队可以在不修改任何 Agent 代码的情况下切换底层模型供应商、调整成本策略或引入新的模型能力。Langfuse 作为可观测性平台，记录每一次 LLM 调用的输入输出、延迟、Token 消耗和成本，使团队能够实时监控 AI 员工的"工作效率"和"工作质量"。在安全层面，每个 Agent 都通过 Authentik 进行身份认证和权限管理，确保它只能访问被授权的资源和执行被许可的操作。Agent 之间的实时协调则通过 Supabase Realtime 实现——当一个 Agent 完成了某项工作并更新了数据库状态，其他相关的 Agent 会通过 WebSocket 立即收到通知并做出响应。最后，Claw Cluster 中的 Agent 还能触发和响应 n8n 自动化工作流，实现与外部系统的集成以及复杂的跨服务编排。

目前 Claw Cluster 仍处于积极开发阶段。团队正在逐步定义每个 Agent 角色的能力边界、权限范围和协作协议。一旦完成部署，Claw Cluster 将成为 EchoThink 从"基础设施平台"向"智能协作平台"跨越的关键里程碑。

## 3. 服务层级与角色分工

理解 EchoThink 的工作流体系，首先需要理解其底层服务的层级架构。整个系统由六个层级组成，每个层级承担特定的职责，层级之间通过标准化的接口和协议相互衔接，共同构成一个完整的协作运行时环境。

最底层是基础设施层，由 PostgreSQL、Redis、MinIO 和 Nginx 四个服务构成。这一层是整个系统的静默骨架，极少直接与终端用户或 AI Agent 交互，但它的稳定性决定了上层所有服务的可靠性。PostgreSQL 16 搭载 pgvector 扩展，作为统一的数据基座，为几乎所有上层服务提供持久化存储，同时支持关系型查询、全文检索和向量相似性搜索。Redis 提供高速缓存和消息队列能力，是 Dify 工作流引擎和 Hatchet 任务调度的关键通信桥梁。MinIO 提供 S3 兼容的对象存储，用于管理所有非结构化数据——图片、文档、代码包、模型权重文件等。Nginx 作为反向代理和 TLS 终端，将所有服务统一暴露在标准化的子域名下，同时处理证书管理和流量路由。

第二层是身份层，由 Authentik 独立承担。Authentik 回答的是"谁可以做什么"这个根本问题。它管理人类用户和 AI Agent 的身份、认证流程、权限分配和 OAuth/OIDC 集成。在 EchoThink 的世界里，一个 AI Agent 和一个人类开发者在身份系统中的表达方式是对等的——它们都有账号、角色、权限范围和可审计的认证记录。Authentik 的 Flow 机制允许团队为不同类型的参与者定义不同的认证路径，例如人类通过 SSO 和多因素认证登录，而 AI Agent 通过 Service Account Token 进行机器对机器的认证。

第三层是数据平台层，由 Supabase 承担。Supabase 不仅仅是一个数据库封装——它提供了自动生成的 RESTful 和 GraphQL API、基于 PostgreSQL Row Level Security 的细粒度访问控制、基于 Elixir/Phoenix 构建的 Realtime 实时通信引擎，以及与身份系统深度集成的文件存储服务。在工作流体系中，Supabase 扮演的角色是"共享状态与实时协调中心"。当一个 AI Agent 更新了某个游戏关卡的设计参数，这个变更会通过 Supabase Realtime 以 WebSocket 消息的形式实时推送给所有订阅了该项目频道的参与者——包括正在 Outline 中撰写相关文档的人类设计师和正在 GitLab 中编写相关代码的另一个 AI Agent。

第四层是 LLM 网关层，由 LiteLLM 承担。LiteLLM 的职责是为整个系统提供统一的 AI 模型访问接口。无论底层使用的是 OpenAI 的 GPT 系列、Anthropic 的 Claude 系列、Google 的 Gemini 系列还是通过 Ollama 运行的本地开源模型，所有 LLM 调用都通过 LiteLLM 的 OpenAI 兼容 API 进行路由。这种抽象使得团队可以根据任务的复杂度、成本敏感度和延迟要求灵活选择模型，同时在一个统一的控制面板上管理 API Key、配额和路由策略。

第五层是 Agent 基础设施层，由 Dify、Hatchet 和 Graphiti 三个服务共同构成。这一层是 AI 工作实际发生的地方。Dify 提供可视化的工作流构建器和运行时执行引擎，允许团队以有向无环图的方式定义 Agent 的行为逻辑，包括 LLM 调用、条件分支、HTTP 请求、代码执行和人工审批节点。Hatchet 在 Dify 之下提供持久化的任务执行基座——当 Dify 工作流分派一个具体任务时，Hatchet 负责将其放入队列、分配给合适的 Worker、处理重试逻辑和超时管理，并在任务完成后收集结果。Graphiti 则与 FalkorDB 图数据库协同工作，为 Agent 提供时序知识图谱能力——不仅记录"现在什么是真的"，还记录"什么时候变成真的"以及"什么时候不再是真的"，使 Agent 具备真正的组织记忆。

最顶层是自动化与应用层，由 n8n、Langfuse、Outline 和 GitLab 构成。这一层是工作成果具象化的地方。n8n 提供低代码自动化工作流，将各种系统事件串联成端到端的自动化管道——例如"当 Outline 文档更新时自动触发 Dify 工作流重新生成相关资产"。Langfuse 提供 LLM 可观测性，让团队清晰地看到每一次 AI 调用的质量、成本和效率。Outline 作为团队知识库，是人类与 AI 共同阅读和编辑文档的空间。GitLab 作为代码托管和 CI/CD 平台，是代码成果最终汇聚和交付的地方。

## 4. 游戏开发团队的典型一天

为了更直观地理解 EchoThink 工作流体系在实际游戏开发中的运作方式，让我们跟随一个使用 EchoThink 的独立游戏团队，走过他们典型的一天。

清晨，当团队成员们陆续打开电脑时，他们做的第一件事不是查看邮件，而是打开 Outline 知识库和 Langfuse 仪表盘。因为在他们睡觉的整个夜晚，Claw Cluster 中的 AI Agent 们一直在工作。叙事设计 Agent 根据昨天会议上确定的新角色设定，在 Outline 中生成了三份备选的角色背景故事草稿，每份都附带了与已有世界观的一致性分析报告。概念美术 Agent 通过 Dify 工作流调用图像生成模型，为新角色产出了十二张概念设计图，全部存储在 MinIO 中并在 Outline 文档里内嵌了预览链接。Code Reviewer Agent 审查了前一天下午提交的六个 Merge Request，在 GitLab 上留下了详细的代码审查意见，其中两个已经被标记为可以合并，另外四个标注了需要人类开发者关注的潜在问题。QA Agent 在夜间运行了完整的回归测试套件，将测试结果和失败分析推送到了 Hatchet 的任务面板上。团队成员一边喝着咖啡，一边浏览这些 AI 在夜间产出的成果，在 Langfuse 仪表盘上检查各 Agent 的 LLM 调用统计——Token 消耗量、平均响应延迟、调用成功率和估算成本，确保 AI 团队的运作效率和成本都在预期范围内。

上午的设计会议中，策划团队在 Outline 中协作编辑游戏设计文档。当他们在文档中更新了某个关卡的难度参数和敌人配置表时，这个文档变更事件通过 n8n 自动化工作流被捕获，并触发了一系列下游动作：Dify 中的关卡验证工作流被自动启动，该工作流首先通过 Graphiti 查询知识图谱以获取相关的游戏平衡性数据和历史设计决策，然后调用 LiteLLM 网关路由到合适的 LLM，分析新参数是否与整体游戏平衡性一致，最后将分析报告以评论的形式自动附加到原始 Outline 文档下方。整个过程在几分钟内完成，设计师们几乎是实时地看到了 AI 对自己设计变更的反馈。

下午进入资产生产阶段。美术总监在 Outline 中批准了 AI 夜间生成的三张概念设计图中的一张，并在文档中添加了修改意见——"色调偏冷，需要增加暖色元素"和"盔甲细节需要更多哥特风格纹理"。这些批注被 n8n 工作流捕获后，触发 Dify 中的资产迭代工作流。该工作流读取原始图片的生成参数，结合美术总监的修改意见重新构造 Prompt，通过 LiteLLM 调用图像生成模型产出新版本的概念图。新生成的图片被存储到 MinIO，同时 Graphiti 中的知识图谱被更新——记录了"角色 X 的概念设计经历了从 V1 到 V2 的迭代，V2 基于美术总监的暖色调和哥特风格反馈生成"这一事实，并将 V1 的状态标记为"已被 V2 取代"。这种时序记录意味着团队在未来任何时候都可以追溯每一个设计决策的演变过程。

与此同时，在代码开发方面，两个 AI 编程 Agent 正在 GitLab 上工作。一个 Agent 根据 Hatchet 分配给它的任务——"实现新关卡的敌人 AI 行为树"——从 Outline 中读取相关的设计规范，从 Graphiti 中查询已有的 AI 行为模式和代码模块信息，然后编写代码并提交 Merge Request。人类开发者收到通知后进行审查，在 GitLab 上与 AI Agent 进行多轮讨论——AI Agent 能够理解审查意见、解释自己的设计选择、根据反馈修改代码并更新 Merge Request。另一个 Agent 正在处理昨天 Code Reviewer Agent 标记的一个性能问题，它分析了相关代码路径，提出了优化方案，并创建了包含性能基准测试的新 Merge Request。所有这些 Agent 的工作过程——它们的 LLM 调用链、工具使用记录、中间推理步骤——都在 Langfuse 中被完整记录，团队的技术负责人可以随时回溯任何一个 Agent 的决策过程。

傍晚时分，团队成员准备离开。但在离开前，策划在 Outline 中更新了明天的工作优先级，QA 负责人在 Hatchet 中设置了夜间需要运行的测试任务，美术总监在文档中留下了对新一批概念图的反馈要求。这些信息都会被 AI Agent 们在夜间读取和处理。当团队成员明早回来时，又会看到一夜之间积累的新成果——代码审查报告、测试分析、新的概念设计、更新的知识图谱。这种人类与 AI 交替工作、持续推进项目的节奏，正是 EchoThink 工作流体系所要实现的核心价值：让创造力不因人类的作息而中断，让项目在二十四小时中都保持前进的动力。

## 5. 从手动到自动：渐进式采纳路径

EchoThink 的工作流体系虽然功能强大且层级丰富，但团队并不需要从第一天起就全面部署所有功能。事实上，EchoThink 的架构设计刻意支持渐进式采纳——团队可以根据自身的技术成熟度和实际需求，分阶段地引入工作流能力，每一个阶段都能独立产生价值，同时为下一阶段的扩展打下基础。

第一阶段建议从 n8n 自动化工作流起步。n8n 提供了直观的可视化工作流编辑器，团队中即使没有深厚编程背景的成员也可以快速上手。在这个阶段，团队可以构建一些简单但实用的自动化流程，例如：当 GitLab 中的 Merge Request 被创建时自动发送通知到团队的通信频道；当 Outline 中的特定文档被更新时自动将变更摘要发送给相关团队成员；定时从外部数据源拉取竞品分析数据并存储到 Supabase 中。这些工作流不涉及 LLM 调用，也不需要 AI Agent 的概念，它们纯粹是事件驱动的自动化管道，但即便如此，它们也能显著减少团队的重复性手工操作，并且让团队熟悉 EchoThink 基础设施的运作方式——Docker Compose 服务管理、Nginx 路由配置、Authentik 身份认证等核心概念都会在这个阶段被自然地接触和理解。

第二阶段引入 Dify AI 工作流。当团队对 n8n 自动化已经驾轻就熟之后，下一步是在工作流中引入 LLM 能力。Dify 的可视化工作流构建器允许团队将 LLM 调用嵌入到结构化的处理流程中，而不是仅仅依赖自由对话。在这个阶段，团队可以构建第一批 AI 增强的工作流：让 LLM 自动为新提交的代码生成文档注释，让 LLM 根据设计文档自动生成测试用例的初始框架，让 LLM 分析会议纪要并提取出行动项和决策点。这些工作流通过 LiteLLM 网关调用模型，通过 Langfuse 记录调用日志，团队开始积累对 AI 能力边界和成本结构的直观认知。同时，Hatchet 开始发挥作用，确保这些 AI 工作流即使在执行过程中遇到模型超时或服务重启等异常情况，也能被可靠地重试和恢复。Graphiti 知识图谱也可以在这个阶段开始积累数据——每一次 AI 工作流的执行结果都被写入图谱，为后续的智能检索和上下文感知奠定数据基础。

第三阶段是全面引入 Claw Cluster 智能员工集群。当团队已经拥有了成熟的自动化管道和经过验证的 AI 工作流模板，并且对 EchoThink 基础设施的运维有了充分的信心之后，就可以部署专业化的 AI Agent 角色，让它们以"结构化员工"的身份参与到日常工作流程中。在这个阶段，AI Agent 不再是被动等待触发的工作流节点，而是主动监控项目状态、发起工作任务、与人类和其他 Agent 实时协调的自主参与者。Authentik 中为每个 Agent 配置了精确的权限边界，Supabase Realtime 为多 Agent 协调提供了实时通信通道，Langfuse 提供了全面的 Agent 行为监控，Hatchet 确保了 Agent 任务的可靠执行。团队的管理重心从"自己做事"转变为"管理和协调一支人机混合团队"，工作效率和产出规模将实现质的飞跃。

这种渐进式路径的核心优势在于降低风险和加速学习。每个阶段的产出都是即时可见的，团队不需要等到整个系统完全部署才能获得价值回报。同时，每个阶段积累的运维经验和工作流模板都会成为下一阶段的基础资产。团队在第一阶段学会的 n8n 工作流构建能力，在第三阶段仍然是 Agent 与外部系统集成的核心手段；在第二阶段打磨的 Dify 工作流模板，在第三阶段会成为 Claw Cluster 中各 Agent 的标准操作程序。这种层层递进、持续复用的采纳路径，确保了团队在拥抱 AI 协作的过程中始终脚踏实地、稳步前进。
