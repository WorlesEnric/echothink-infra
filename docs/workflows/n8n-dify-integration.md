# n8n 与 Dify 深度集成指南

## 双平台协作的设计理念

在 EchoThink 基础设施中，n8n 和 Dify 分别承担着截然不同却彼此互补的职责，二者的协作构成了整个游戏开发 AI 自动化体系的核心骨架。许多团队在初次接触这套架构时会产生一个自然的疑问：既然 Dify 本身已经具备工作流编排能力，为什么还需要 n8n？反过来，n8n 也可以通过 HTTP 节点直接调用大语言模型的 API，那 Dify 的价值又体现在哪里？要回答这个问题，我们需要从两个平台各自的设计哲学说起。

n8n 是一个事件驱动的自动化编排引擎，它的核心能力在于连接外部服务、响应各类事件触发器、以及对数据进行清洗和转换。在我们的游戏开发工作流中，n8n 负责监听 GitLab 的 Merge Request 事件、接收 Outline 知识库的文档变更 Webhook、定时轮询 Hatchet 任务队列的状态、向 MinIO 上传和下载美术资源文件、以及与 Supabase 数据库进行读写交互。可以说，n8n 是整个系统的"神经系统"，它感知外部世界的变化，将各个服务节点的数据流串联起来，确保信息在正确的时间被传递到正确的地方。n8n 的节点生态极其丰富，几乎所有主流 SaaS 服务和开发工具都有对应的集成节点，这意味着团队可以在不编写任何代码的情况下，通过可视化拖拽的方式完成复杂的跨服务编排。

Dify 则是一个专注于 AI 推理与智能体编排的平台，它的核心价值在于 Prompt 工程管理、RAG 知识检索增强、Agent 工具调用、以及多模型路由。在游戏开发场景下，Dify 承载着所有需要"思考"的任务：分析策划文档并生成角色技能描述、审查代码架构提案并给出改进建议、根据美术风格指南评估素材的一致性、将零散的会议记录整理为结构化的设计决策文档。Dify 是整个系统的"大脑"，它不关心数据从哪里来、结果要送到哪里去，它只专注于利用大语言模型的推理能力对输入信息进行深度处理并产出高质量的输出。Dify 提供了完整的 Prompt 版本管理、A/B 测试、对话日志审计等能力，这些功能对于持续优化 AI 生成内容的质量至关重要。

当 n8n 和 Dify 协同工作时，系统的能力远超二者各自的总和。n8n 负责"何时做"和"做完之后怎么办"，Dify 负责"怎么想"和"想出什么"。一个典型的游戏开发工作流可能是这样的：策划在 Outline 中更新了一份 BOSS 战设计文档，n8n 通过 Webhook 感知到这个变化，提取文档内容后调用 Dify 的工作流进行分析，Dify 利用 RAG 检索相关的历史设计文档和平衡性数据，通过 LiteLLM 网关调用大语言模型生成影响分析报告，然后 n8n 接收 Dify 的输出，自动在 GitLab 创建相关的开发任务 Issue，同时将分析报告写回 Outline 供团队成员评审。整个过程无需人工介入，从文档更新到任务创建只需要数十秒。这种将"自动化编排"与"AI 推理"分离的架构设计，不仅让每个平台都能发挥各自的最大优势，也使得系统的可维护性和可扩展性大大提升。

## 模式一：n8n 调用 Dify（n8n 编排，Dify 执行 AI 推理）

这是最基础也是最常用的集成模式。在这种模式下，n8n 作为主控编排器，通过 HTTP Request 节点调用 Dify 暴露的 API 接口，将需要 AI 处理的数据发送给 Dify，等待 Dify 完成推理后获取结果，再继续后续的自动化流程。这种模式适用于所有"先收集数据、再调用 AI、最后分发结果"的线性工作流场景。

在 EchoThink 的内部网络中，Dify API 服务运行在 `dify-api:5001` 地址上，n8n 可以直接通过 Docker 内部 DNS 访问该服务，无需经过 Nginx 反向代理层。Dify 提供两类核心 API 接口：对话型接口（Chat Messages API）和工作流运行接口（Workflows Run API）。对话型接口适用于需要多轮交互或利用 Dify 内置 Agent 能力的场景，而工作流运行接口则适用于输入输出明确的单次处理任务。

当游戏策划需要为一个新设计的 BOSS 角色生成技能描述时，n8n 可以通过以下方式调用 Dify 的对话型接口：

```
POST http://dify-api:5001/v1/chat-messages
Authorization: Bearer app-xxxxxxxxxxxxx
Content-Type: application/json

{
  "inputs": {},
  "query": "为这个BOSS角色生成三个技能描述",
  "response_mode": "blocking",
  "user": "n8n-workflow-001"
}
```

这个请求中有几个关键参数需要理解。`Authorization` 头部的值是 Dify 应用的 API 密钥，每个 Dify 应用都有独立的密钥，格式为 `app-` 前缀加上一串随机字符。`response_mode` 设置为 `blocking` 意味着 Dify 会在完成所有推理之后才返回完整的响应，这对于 n8n 的 HTTP Request 节点来说是最简单的处理方式，因为节点会等待响应返回后才继续执行下一步。如果将 `response_mode` 设置为 `streaming`，Dify 会以 Server-Sent Events 的格式逐步返回生成内容，这种方式适合需要实时展示生成进度的前端应用，但在 n8n 的后台自动化场景中通常不需要。`user` 字段用于标识调用者身份，在 Langfuse 的追踪日志中可以根据这个字段筛选特定工作流产生的调用记录。

对于输入输出结构更明确的场景，比如根据角色名称和风格要求生成角色描述，使用 Dify 的工作流运行接口会更加合适：

```
POST http://dify-api:5001/v1/workflows/run
Authorization: Bearer app-xxxxxxxxxxxxx
Content-Type: application/json

{
  "inputs": {"character_name": "暗影刺客", "style": "dark fantasy"},
  "response_mode": "blocking",
  "user": "n8n-asset-pipeline"
}
```

工作流运行接口与对话型接口的核心区别在于：它通过 `inputs` 对象传递所有输入变量，这些变量与 Dify 工作流编辑器中定义的"开始节点"输入变量一一对应。没有 `query` 字段，因为工作流的执行逻辑完全由 Dify 内部的节点编排决定。这种方式的优势在于接口契约非常清晰，n8n 端和 Dify 端的开发者可以独立工作，只需要约定好输入变量的名称和格式即可。

在 n8n 中处理 Dify 的响应时，需要注意响应的 JSON 结构。对话型接口的响应中，生成内容位于 `$json.answer` 字段；而工作流运行接口的响应中，输出结果位于 `$json.data.outputs` 对象下，具体的字段名称取决于 Dify 工作流中"结束节点"定义的输出变量。n8n 的 Set 节点或 Code 节点可以用来提取和重组这些数据，为后续节点提供所需的输入格式。

在实际的游戏开发流水线中，一个完整的 n8n 工作流可能包含多次 Dify 调用。例如，角色资源生成流水线可能先调用一次 Dify 工作流生成角色的文字描述，然后调用另一次 Dify 工作流将文字描述转化为 Stable Diffusion 的 Prompt，最后由 n8n 将生成的 Prompt 发送给图像生成服务。每一次 Dify 调用都是独立的、无状态的，这使得整个流水线的每个环节都可以独立调试和优化。

## 模式二：Dify 回调 n8n（Dify Agent 使用 n8n Webhook 作为外部工具）

第二种集成模式的控制方向与第一种恰好相反：Dify 的 Agent 在推理过程中主动调用 n8n 暴露的 Webhook 接口，以执行外部操作。这种模式充分发挥了 Dify Agent 的自主决策能力，让 AI 自行判断何时需要与外部系统交互，而不是由 n8n 预先编排好所有步骤。

要理解这种模式的价值，可以想象这样一个场景：游戏策划在 Dify 的对话界面中与一个"项目管理助手" Agent 交流，讨论一个新功能的实现方案。在对话过程中，Agent 可能判断需要在 GitLab 上创建一个 Issue 来追踪这个功能的开发进度，或者需要更新 Outline 知识库中的技术规范文档，又或者需要向特定的团队成员发送通知。这些操作涉及与外部服务的交互，而 Dify 本身并不直接支持 GitLab 或 Outline 的 API 调用。此时，n8n 的 Webhook 节点就成为了完美的桥梁。

配置这种模式需要两个步骤。首先，在 n8n 中创建一个或多个 Webhook 工作流，每个工作流处理特定类型的外部操作。然后，在 Dify 中将这些 Webhook URL 注册为 Agent 的自定义工具。当 Agent 在推理过程中决定需要执行某项外部操作时，它会自动调用对应的工具，触发 n8n 的 Webhook 工作流。

n8n 端的 Webhook 工作流设计通常采用"统一入口、内部路由"的模式。通过一个 Webhook 节点接收所有来自 Dify 的回调请求，然后根据请求体中的 `action` 字段使用 Switch 节点路由到不同的处理分支：

```
n8n Webhook URL: http://n8n:5678/webhook/dify-callback
Method: POST
Body: {
  "action": "create_gitlab_issue",
  "project": "game-core",
  "title": "{{$json.title}}",
  "description": "{{$json.description}}"
}
```

在这个示例中，n8n 的 Webhook 节点监听 `/webhook/dify-callback` 路径。当收到请求后，Switch 节点检查 `action` 字段的值：如果是 `create_gitlab_issue`，则路由到 GitLab Issue 创建分支；如果是 `update_outline_doc`，则路由到 Outline 文档更新分支；如果是 `send_notification`，则路由到通知发送分支。每个分支可以包含任意复杂的自动化逻辑，因为 n8n 拥有完整的服务集成生态。

在 Dify 端配置自定义工具时，需要在 Agent 应用的"工具"设置中添加一个 API 工具。工具的 Schema 定义了 Agent 可以传递的参数，比如 `action`、`project`、`title`、`description` 等字段。Dify 会根据 Schema 的描述信息让 Agent 理解每个工具的用途和参数含义，从而在对话中自动决定何时调用哪个工具。这种方式的优雅之处在于，工具的调用完全由 AI 根据对话上下文自主决定，无需人工预先规划调用时机。

需要注意的是，由于 n8n 和 Dify 都运行在 EchoThink 的内部 Docker 网络 `echothink-internal` 上，Webhook URL 应当使用内部 DNS 地址 `http://n8n:5678`，而不是外部的 `https://n8n.${DOMAIN}`。使用内部地址不仅避免了不必要的 Nginx 反向代理开销和 TLS 握手延迟，更重要的是确保了这些回调请求不会暴露到公网，从而提升了整个系统的安全性。如果团队后续将 n8n 和 Dify 部署在不同的网络或不同的主机上，则需要切换为外部 URL 并配置相应的认证机制。

这种回调模式在游戏开发中的一个典型应用是"智能代码审查助手"。当开发者在 Dify 对话界面中粘贴一段代码请求审查时，Agent 不仅会分析代码质量，还可能主动调用 n8n Webhook 在 GitLab 上搜索相关的代码文件以获取更多上下文，查询 Outline 中的编码规范文档来确保审查意见与团队标准一致，甚至直接创建 GitLab Issue 来记录发现的问题。

## 模式三：双向嵌套（复合工作流）

第三种模式是前两种模式的融合与升华，它构建了一个 n8n 与 Dify 之间的双向调用链路，实现了真正意义上的"人机协作闭环"。在这种模式下，n8n 触发 Dify 工作流进行 AI 推理，Dify 的 Agent 在推理过程中通过 Webhook 回调 n8n 执行外部操作，n8n 处理完毕后返回结果给 Dify，Dify 基于新获取的信息继续推理，最终 n8n 接收 Dify 的完整输出并完成后续的自动化操作。这种层层嵌套的双向通信，使得 AI 能够在推理过程中动态获取外部信息，而不是仅仅依赖初始输入。

以一个具体的游戏开发场景来说明这种模式的强大之处：假设团队的策划在 Outline 知识库中更新了一份"BOSS 战斗机制设计文档"。从文档更新到最终产出可执行的开发任务，整个流程涉及五个阶段。第一阶段，Outline 通过 Webhook 通知 n8n 有文档被更新，n8n 接收事件并提取文档的完整内容。第二阶段，n8n 将文档内容发送给 Dify 的"设计变更分析"工作流，请求 Dify 分析这次设计变更的影响范围。第三阶段，Dify 的 Agent 在分析过程中意识到需要了解当前代码库中与 BOSS 战斗相关的模块结构，于是通过 Webhook 回调 n8n，由 n8n 查询 GitLab API 获取相关代码文件的目录结构和最近的提交记录，并将结果返回给 Dify。第四阶段，Dify 结合策划文档的变更内容和代码库的现状，生成一份详尽的影响分析报告，包括需要修改的代码模块、潜在的平衡性风险、以及建议的实现方案。第五阶段，n8n 接收 Dify 的分析报告，自动在 GitLab 中为每个需要修改的模块创建独立的 Issue，将影响分析报告写入 Outline 作为原始设计文档的子文档，并通过通知渠道提醒相关开发者查看新任务。

实现这种双向嵌套工作流需要特别注意几个技术要点。首先是会话状态的管理：当 Dify 回调 n8n 时，n8n 需要在响应中返回查询结果，而 Dify 需要能够将这个结果纳入后续的推理上下文。在 Dify 的 Agent 模式下，工具调用的返回值会自动成为对话历史的一部分，Agent 可以基于工具返回的信息继续推理，因此这一点是天然支持的。其次是超时配置：由于双向调用链路较长，整个流程的执行时间可能超过单次 HTTP 请求的默认超时时间。n8n 的 HTTP Request 节点需要将超时设置调大（建议至少 300 秒），同时 Nginx 反向代理的 `proxy_read_timeout` 也需要相应调整，在当前的 n8n Nginx 配置中已经设置为 300 秒。最后是错误处理：任何一个环节的失败都可能导致整个链路断裂，因此每个 n8n 工作流都应该配置 Error Trigger 节点，在出错时记录日志并发送告警通知。

这种复合工作流模式代表了 EchoThink 平台最高级别的自动化能力。它不仅仅是简单的"触发-执行"，而是一个 AI 能够主动探索外部世界、获取所需信息、综合分析并给出行动建议的完整智能循环。对于游戏开发团队来说，这意味着许多原本需要项目经理花费数小时手动完成的"需求拆解-影响评估-任务分配"流程，现在可以在几分钟内由 AI 自动完成。

## 认证与安全配置

在 EchoThink 的内部网络架构中，n8n 和 Dify 之间的通信安全建立在多层防护机制之上。理解并正确配置这些安全措施，是确保整个自动化体系可靠运行的基础。

首先是 API 密钥的管理。Dify 为每个应用生成独立的 API 密钥，格式为 `app-` 前缀加上随机字符串。这些密钥应当存储在 n8n 的凭证管理系统中，而不是硬编码在工作流的节点参数里。在 n8n 中创建一个类型为"Header Auth"的凭证，将名称设置为 `Authorization`，值设置为 `Bearer app-xxxxxxxxxxxxx`，然后在 HTTP Request 节点中引用这个凭证。这样做的好处是密钥只在 n8n 的加密存储中保存一份，所有引用该凭证的工作流都会自动获取最新的密钥值，当需要轮换密钥时只需在凭证管理中修改一处即可。n8n 使用 `N8N_ENCRYPTION_KEY` 环境变量对所有凭证进行 AES 加密存储，这个密钥在 Docker Compose 配置中通过环境变量注入，确保凭证数据即使被直接访问数据库也无法被读取。

其次是网络层面的隔离。从 Docker Compose 的配置可以看到，EchoThink 使用了双网络架构：`echothink-internal` 网络用于服务间的内部通信，`echothink-public` 网络用于需要对外暴露的服务。Dify API 服务同时连接了两个网络，因为它既需要接收来自 n8n 的内部调用，也需要通过 Nginx 对外提供 Web 界面。n8n 同样连接了两个网络。当 n8n 调用 Dify 时，请求通过 `echothink-internal` 网络直接到达 `dify-api:5001`，完全绕过 Nginx 和 TLS 层，这不仅降低了延迟，也避免了在内部通信中引入不必要的证书管理复杂度。当 Dify 的 Agent 回调 n8n 的 Webhook 时，同样通过内部网络直接访问 `n8n:5678`，无需经过外部路由。

第三个安全考量是 Dify Sandbox 的隔离。Dify 包含一个代码执行沙箱服务 `dify-sandbox`，用于安全地执行工作流中的自定义代码节点。这个沙箱通过独立的 API 密钥 `DIFY_SANDBOX_API_KEY` 进行认证，并且设置了 15 秒的工作超时 `WORKER_TIMEOUT`，防止恶意或错误的代码无限运行。沙箱虽然启用了网络访问（`ENABLE_NETWORK: true`），但仅连接到 `echothink-internal` 网络，无法直接访问外部互联网。如果工作流中的代码节点需要访问外部 API，应当通过 n8n 的 Webhook 回调模式间接实现，由 n8n 负责外部调用并返回结果。

最后，对于生产环境的部署，建议为 Dify 的不同应用使用不同的 API 密钥，并在 LiteLLM 网关层面为每个密钥配置独立的速率限制和预算上限。这样即使某个工作流因为逻辑错误导致了无限循环的 API 调用，LiteLLM 的速率限制也能在预算耗尽之前自动中断调用链路，防止产生高额的 API 费用。Langfuse 的可观测性平台会实时记录所有通过 LiteLLM 路由的模型调用，团队可以通过 Langfuse 的仪表盘监控每个工作流的调用频率、Token 消耗和响应延迟，及时发现异常情况。

## Claw Cluster 作为工作流触发者

Claw Cluster 是 EchoThink 生态系统中即将上线的核心组件，它将彻底改变 n8n 和 Dify 工作流的触发方式。在当前的架构中，工作流的触发主要依赖两种机制：外部事件的 Webhook 通知（如 GitLab 的代码推送事件、Outline 的文档更新事件）和定时调度（如每小时执行一次的数据同步任务）。这些触发方式虽然有效，但本质上是被动的和机械的——它们不理解项目的当前状态，也无法根据全局上下文做出智能的调度决策。

Claw Cluster 的设计目标是成为一个自主的工作流编排者，它由多个协同工作的 AI Agent 组成，每个 Agent 扮演一个特定的项目角色（如技术主管、测试工程师、美术总监等）。这些 Agent 持续监控项目的整体状态，包括 GitLab 中的代码变更和 Issue 进度、Outline 中的设计文档更新、Supabase 中的项目数据指标、以及 Graphiti 知识图谱中的实体关系变化。基于对这些多维信息的综合分析，Claw Cluster 会主动判断当前哪些工作流需要被触发。

从技术实现的角度来看，Claw Cluster 将通过两个主要接口与现有的 n8n 和 Dify 服务进行交互。对于 n8n，Claw Cluster 会通过 n8n 的 Webhook URL 触发预定义的自动化工作流，传入经过 AI 分析后确定的参数。例如，当 Claw Cluster 的"技术主管" Agent 分析代码库后发现某个模块的技术债务已经累积到需要重构的程度时，它可以触发一个 n8n 工作流来自动创建重构任务的 Issue 并通知相关开发者。对于 Dify，Claw Cluster 会直接调用 Dify 的 API 来执行需要深度 AI 推理的任务，如代码质量评估、设计一致性审查、或性能瓶颈分析。

Claw Cluster 与现有工作流的关键区别在于触发的"智能性"。传统的 Webhook 触发是"事件发生了，所以执行工作流"的简单因果关系；而 Claw Cluster 的触发逻辑是"我分析了项目的当前状态，认为现在应该执行这个工作流，并且这些参数是最合适的"。这种智能触发能力意味着团队不再需要预先枚举所有可能的自动化场景并为每个场景配置触发规则——Claw Cluster 会根据项目的实际需求自适应地决定何时执行哪些操作。

在游戏开发的具体场景中，Claw Cluster 的价值将尤为突出。游戏项目的开发周期中充满了跨职能的协作需求：策划更新了关卡设计，美术需要相应调整场景素材，程序需要修改关卡加载逻辑，QA 需要更新测试用例。在传统流程中，这些跨职能的协调工作依赖项目经理的人工跟进，信息传递的延迟和遗漏是常态。Claw Cluster 将通过持续监控各个职能的工作产出，自动识别跨职能的依赖关系，并通过触发 n8n 和 Dify 的工作流来确保信息在正确的时间被传递到正确的人手中。它不仅仅是一个触发器，更是一个理解项目全局的智能协调者，真正实现了"AI 作为团队成员"的愿景。

当 Claw Cluster 正式上线后，当前所有通过 Webhook 和定时调度触发的工作流仍然会保持运行，Claw Cluster 将作为一个额外的、更智能的触发源加入到现有体系中。团队可以逐步将部分依赖人工判断的触发场景迁移到 Claw Cluster，同时保留原有的事件驱动触发作为兜底机制，确保在 Claw Cluster 做出错误判断时，关键的自动化流程不会中断。
