# LiteLLM — EchoThink 的大模型统一网关

## 概述与定位

在 EchoThink 这样一个以 AI 智能体为核心生产力的平台中，大语言模型的调用是最基础也最频繁的操作之一。无论是 Dify 中编排的 AI 工作流需要调用 Claude 来生成游戏剧情对白，还是 Graphiti 知识图谱引擎需要利用嵌入模型将游戏设计文档向量化，又或者是开发人员通过代码助手请求代码补全和审查，所有这些场景都涉及对不同供应商、不同能力层级的大语言模型的调用。如果每个服务都直接对接各个模型供应商的 API，不仅会造成 API Key 的分散管理和安全隐患，还会导致成本追踪困难、故障难以排查、模型切换时需要修改多处代码等一系列运维噩梦。LiteLLM 正是为了解决这些问题而存在——它作为 EchoThink 平台的大模型统一网关，将所有 LLM 调用收束到一个单一入口点，对外提供完全兼容 OpenAI API 格式的统一接口，对内则根据策略将请求智能路由到最合适的模型。

LiteLLM 在 EchoThink 中以自定义构建的 Docker 镜像运行，容器名称为 `echothink-litellm`，监听 4000 端口并通过命令行参数加载 `/app/config.yaml` 配置文件。它同时连接 `echothink-internal`（用于接收来自 Dify、Graphiti 等内部服务的请求）和 `echothink-public`（通过 Nginx 反向代理对外暴露管理 API）两个 Docker 网络。LiteLLM 依赖 PostgreSQL 数据库（数据库名称 `litellm`，用户 `litellm_user`）来持久化存储模型使用记录、费用追踪数据和用户配额信息，同时利用 Redis 实现响应缓存和分布式速率限制，两者都通过健康检查确保在 LiteLLM 启动前就绪。

## 模型分组与智能路由

LiteLLM 的核心价值在于其模型分组（Model Group）机制。EchoThink 的配置中定义了五个功能明确的模型组，每个组包含来自不同供应商的多个模型，形成了一个既有冗余保障又能根据场景灵活选择的模型矩阵。

第一个是 `high-quality` 组，顾名思义，这个组汇集了当前市场上推理能力最强的模型，包括 Anthropic 的 Claude Opus 4.6 和 OpenAI 的 GPT-4o。当任务需要深度推理、复杂分析或高质量创意生成时——例如为一个开放世界 RPG 设计环环相扣的主线剧情，或者分析一个复杂游戏系统的平衡性——调用方只需将请求发送到 `high-quality` 这个模型名称，LiteLLM 会根据延迟和可用性自动选择其中一个模型来处理。

第二个是 `cost-effective` 组，包含 Claude Haiku 4.5 和 GPT-4o-mini。这两个模型在保持相当不错的生成质量的同时，其调用成本仅为顶级模型的几分之一。对于那些需要大量调用但不需要最高推理能力的场景——例如批量生成 NPC 的闲聊对话、翻译多语言 UI 文案、或者对大量游戏日志进行初步分类——这个组是最经济的选择。

第三个是 `fast` 组，同样由 Claude Haiku 4.5 和 GPT-4o-mini 组成，但其路由策略侧重于延迟优化。在游戏开发的实时交互场景中——比如开发者在 IDE 中等待代码补全建议，或者 QA 工具需要快速判断一个 bug 报告的优先级——响应速度比回答的深度更重要，这时 `fast` 组确保请求被路由到当前延迟最低的模型。

第四个是 `code` 组，专门为代码生成和分析任务设计，包含 Claude Sonnet 4.6 和 GPT-4o。Claude Sonnet 4.6 在代码理解和生成方面表现卓越，非常适合游戏开发中的常见编程任务：编写游戏逻辑、重构渲染管线代码、生成单元测试、审查合并请求中的代码变更。GPT-4o 作为后备，在 Sonnet 不可用时接替其工作。

第五个是 `embedding` 组，包含 OpenAI 的 text-embedding-3-small 和 text-embedding-3-large 两个嵌入模型。嵌入模型虽然不生成文本，但在 EchoThink 平台中同样不可或缺——Graphiti 知识图谱的构建依赖嵌入向量来计算文档之间的语义相似度，Dify 中的 RAG（检索增强生成）流程需要将知识库文档转换为向量以支持语义搜索。Small 变体提供了性价比最优的嵌入方案，Large 变体则在需要更高维度和更精确语义表示时发挥作用。

## 延迟路由与自动故障转移

LiteLLM 的路由引擎采用基于延迟的路由策略（`latency-based-routing`），这意味着在同一个模型组内，系统会持续监控每个模型的响应延迟，并优先将新请求路由到当前延迟最低的模型实例。这种策略在模型供应商出现间歇性延迟波动时尤其有效——当 OpenAI 的 API 因为流量高峰出现响应变慢时，系统会自动将更多请求导向 Anthropic 的模型，反之亦然，无需任何人工干预。

更重要的是故障转移（Fallback）机制的设计。EchoThink 的配置定义了一套精心设计的降级链路：当 `high-quality` 组中的模型全部不可用时，请求会首先尝试降级到 `code` 组，然后再降级到 `cost-effective` 组；`code` 组不可用时降级顺序为 `high-quality` 然后 `cost-effective`；`cost-effective` 组降级到 `fast` 组；`fast` 组降级到 `cost-effective` 组。这种交叉降级设计确保了在任何单一供应商完全宕机的极端情况下，平台的 AI 能力不会完全中断——系统会用可用的替代模型继续提供服务，尽管质量或速度可能有所折中。每个请求最多重试 3 次，重试间隔为 5 秒，单次请求超时时间为 120 秒。当一个模型连续失败 3 次后会进入 60 秒的冷却期，在此期间不会再向其发送新请求。

## Redis 缓存与成本优化

LiteLLM 集成了 Redis 缓存机制，使用独立的 `litellm_cache` 命名空间存储缓存数据。当两个不同的服务（或者同一服务的两次调用）使用完全相同的提示词和参数请求同一个模型时，第二次调用会直接返回缓存结果而不会实际发起一次新的 API 调用。这在游戏开发的很多场景中都能带来可观的成本节约。例如，当多个团队成员在 Dify 中使用同一个 AI 工作流处理相似的输入时，或者当自动化流水线重复执行相同的代码分析任务时，缓存机制避免了不必要的重复计费。`drop_params` 设置为 true 意味着 LiteLLM 会自动丢弃目标模型不支持的请求参数，而不是返回错误——这在多模型路由场景下非常实用，因为不同模型支持的参数集合并不完全一致。

LiteLLM 通过 PostgreSQL 数据库持久化存储所有的使用记录和成本数据。`allow_user_auth` 设为 true 允许基于用户维度的认证和计量，`store_model_in_db` 设为 true 则将模型配置也同步存储到数据库中，使得可以通过管理 API 动态添加或修改模型配置而无需重启服务。这种设计为运营团队提供了完整的成本可见性——可以精确地知道每个服务、每个用户、每个模型组在任意时间段内的 Token 消耗和对应费用，从而做出数据驱动的资源分配决策。

## 游戏开发场景中的应用

对于专注游戏设计与开发的团队来说，LiteLLM 的统一网关架构将 AI 的使用门槛降到了最低。策划人员通过 Dify 编排工作流时，只需要关心自己的任务需要"高质量"还是"性价比"的模型，而不需要了解 Claude 和 GPT 之间的 API 差异；程序员在集成 AI 辅助编码工具时，只需将 API 地址指向 LiteLLM 的 4000 端口并使用 OpenAI 兼容格式发送请求，底层模型的选择和切换完全由网关透明处理。

在具体的游戏开发任务中，不同的模型组承担着不同的角色。`high-quality` 组适合处理游戏世界观构建、主线剧情编写、复杂系统设计分析等需要深度思考的任务。`code` 组负责支撑日常的代码生成、重构建议和代码审查工作，Claude Sonnet 4.6 在理解游戏引擎代码结构和生成符合项目风格的代码方面表现尤为出色。`cost-effective` 组则承担起大量的批处理工作：为数百个 NPC 生成各具特色的简介文本、将设计文档翻译成多种语言、从玩家反馈中提取关键问题等。`embedding` 组支撑着知识管理的基础设施，将散落在各处的游戏设计文档、美术风格指南、技术规范转化为可以被语义搜索的向量表示。

## Claw Cluster 的智能调度中枢

即将上线的 Claw Cluster（AI 员工集群）将使 LiteLLM 的统一网关架构发挥出最大价值。作为 EchoThink 的智能核心，Claw Cluster 中所有 AI 智能体的 LLM 调用都将强制通过 LiteLLM 进行路由，这一设计决策带来了三重关键优势。

首先是统一的成本管控。当 Claw Cluster 中可能同时运行数十个 AI 智能体——有负责概念美术描述生成的 Artist Agent，有负责代码审查和自动修复的 Engineer Agent，有负责游戏文案撰写的 Writer Agent，有负责自动化测试的 Tester Agent——如果每个 Agent 都直接调用模型供应商的 API，成本追踪将变成一场噩梦。通过 LiteLLM，所有调用汇聚到一个节点，每一笔 Token 消耗都被精确记录并关联到发起调用的具体 Agent 和任务，使得团队能够清晰地了解 AI 的运营成本结构，并据此优化资源分配。

其次是统一的可观测性。LiteLLM 与 Langfuse（EchoThink 的 LLM 可观测性平台）深度集成，每一次 LLM 调用的完整生命周期——包括提示词内容、模型选择、响应延迟、Token 消耗和输出质量——都被记录和追踪。当 Claw Cluster 中的某个 Agent 产出了不符合预期的结果时，团队可以通过 Langfuse 精确定位到具体的 LLM 调用，分析是提示词设计的问题还是模型选择的问题，从而持续改进 AI 智能体的表现。

最后是统一的速率管理和优雅降级。模型供应商通常对 API 调用施加速率限制，当 Claw Cluster 的多个 Agent 同时发起大量请求时，如果没有集中管控，很容易触发限流而导致任务失败。LiteLLM 在网关层面统一管理速率配额，在接近限制时自动进行流量整形和请求排队，并在必要时触发跨供应商的故障转移，确保 Claw Cluster 的整体运转不会因单一供应商的限流而中断。

## 关键配置参考

LiteLLM 服务端口：4000

核心环境变量：
- `LITELLM_MASTER_KEY` — 管理 API 的访问密钥
- `LITELLM_DATABASE_URL` — PostgreSQL 连接字符串
- `OPENAI_API_KEY` — OpenAI API 密钥
- `ANTHROPIC_API_KEY` — Anthropic API 密钥
- `REDIS_HOST` — Redis 主机（默认 `redis`）
- `REDIS_PORT` — Redis 端口（默认 `6379`）

模型组列表：`high-quality`、`cost-effective`、`fast`、`code`、`embedding`

路由策略：`latency-based-routing`，3 次重试，5 秒重试间隔，120 秒超时

配置文件位置：`services/litellm/config.yaml`
