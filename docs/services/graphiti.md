# Graphiti — EchoThink 的时序知识图谱

## 平台定位与核心价值

Graphiti 0.22.0 与 Neo4j 5 Community 共同构成了 EchoThink 基础设施中的时序知识图谱层。在传统的开发团队中，知识通常散落在文档、聊天记录、会议纪要和个人笔记之中，随着项目推进和人员变动，关键决策的上下文逐渐丢失，设计理念的演变历程无人知晓。Graphiti 的引入正是为了解决这个根本性问题——它不仅存储知识的当前状态，还记录知识随时间的演变过程，让团队（以及 AI Agent）能够理解"为什么事情变成了现在这样"。

对于一支从事游戏设计与开发的团队而言，知识图谱的价值尤为突出。游戏世界本身就是一个由角色关系、物品依赖、剧情脉络、地图连接和机制交互编织而成的复杂网络。传统的文档和数据库难以自然地表达"勇者之剑是由铸剑师在第三章中用龙鳞锻造而成"这样的多实体关联关系，而图数据库恰好以节点和边的方式天然支持这种表达。Graphiti 在此基础上增加了时间维度，使得团队可以追踪"这个角色的设定在什么时候、因为什么原因被修改过"，为复杂游戏项目中的设计一致性提供了强有力的保障。

## 容器架构与服务组成

Graphiti 在 EchoThink 中由两个容器协同工作。Neo4j 5 Community 作为底层图数据库引擎，提供高性能的图数据存储和查询能力。Graphiti Server 作为上层应用服务，提供面向业务的 HTTP API，并集成了 LLM 驱动的实体提取和关系构建能力。

Neo4j 容器（echothink-neo4j）运行 Neo4j 5 Community Edition，配置了精心调优的内存参数以在资源有限的环境中保持良好性能。初始堆内存设置为 256MB（`NEO4J_server_memory_heap_initial__size`），最大堆内存 512MB（`NEO4J_server_memory_heap_max__size`），页面缓存 256MB（`NEO4J_server_memory_pagecache_size`），整体内存限制为 4GB（通过 Docker deploy resources 配置）。数据持久化到 `neo4j-data` 卷中，仅连接 echothink-internal 网络，不直接对外暴露。Neo4j 的健康检查通过 wget 探测 7474 端口的 HTTP 接口，由于图数据库启动时需要加载索引和恢复事务日志，启动等待时间设置了较为宽裕的 360 秒（6 分钟），最大重试 20 次，确保在大型数据集的情况下也能等待启动完成。

Graphiti Server 容器运行 `zepai/graphiti:0.22.0` 镜像，通过 Bolt 协议（`bolt://neo4j:7687`）连接 Neo4j 数据库，在 8000 端口上提供 HTTP API 服务。它依赖 Neo4j 容器通过健康检查后才会启动，同时连接到 echothink-internal 和 echothink-public 两个网络，使得内部服务和外部用户都可以访问知识图谱 API。健康检查端点位于 `/healthcheck`，启动等待时间 30 秒，每 30 秒检查一次。

## LLM 驱动的智能知识构建

Graphiti 最引人注目的特性在于它能够利用大语言模型自动从非结构化文本中提取实体和关系，构建结构化的知识图谱。当一段游戏设计文档、一份会议纪要或一条聊天消息被输入 Graphiti 时，内置的 NLP 管线会调用 LLM 识别其中的实体（角色、地点、物品、概念等）以及它们之间的关系（属于、位于、创造、克制等），然后将这些信息以节点和边的形式写入 Neo4j 图数据库。

Graphiti Server 通过配置 `OPENAI_API_BASE` 和 `OPENAI_BASE_URL` 指向 `http://litellm:4000/v1`，将所有模型请求路由到 EchoThink 的 LiteLLM 网关。它使用名为 `cost-effective` 的模型配置进行实体提取和关系推理（`MODEL_NAME: cost-effective`），使用名为 `embedding` 的嵌入模型进行向量化表示（`EMBEDDING_MODEL_NAME: embedding`）。这种通过 LiteLLM 统一管理模型的方式使得团队可以根据成本和质量需求灵活切换底层模型，而无需修改 Graphiti 的配置。

这种 LLM 驱动的知识提取方式大幅降低了知识图谱的构建门槛。传统的知识图谱需要领域专家手动标注实体和关系，成本高昂且难以扩展。Graphiti 让这个过程自动化，团队成员只需正常撰写文档和记录决策，系统就能自动将这些非结构化信息转化为结构化的知识网络。

## 时序感知与知识演变追踪

Graphiti 的名称中"时序"（temporal）二字并非装饰，而是其核心设计理念的体现。每一条知识（无论是实体属性还是关系）都带有时间戳，记录了它被创建或修改的时间。当一个角色的设定被更新时，Graphiti 不会简单地覆盖旧数据，而是保留变更历史，使得团队可以查询任意时间点上的知识状态。这种时序感知在游戏开发中有着丰富的应用场景。

在一个持续迭代的游戏项目中，角色的能力值可能经历过多次平衡调整，剧情线可能在不同版本中有过重大变更，关卡难度可能根据玩家反馈多次修正。Graphiti 的时序知识图谱忠实记录了这些变化的轨迹，当团队需要理解"为什么这个 Boss 的血量从 10000 降到了 5000"时，可以在知识图谱中追溯到做出这个决策的时间点以及相关的上下文信息。这种能力对于维护设计一致性、避免重复讨论已决议的问题、以及帮助新成员快速了解项目演变历史都具有重要价值。

## 游戏开发场景中的应用

游戏世界知识图谱是 Graphiti 在游戏开发中最直接的应用场景。团队可以将整个游戏世界的设定信息——角色关系网络、物品合成树、技能克制关系、地图区域连接、任务前置条件依赖——全部建模为知识图谱。这个图谱不仅服务于人类设计师查阅参考，更重要的是为 AI Agent 提供了理解游戏世界的结构化知识基础。当 AI 需要生成一段新的剧情对话时，它可以先查询知识图谱中相关角色之间的关系和历史互动，确保生成的内容与已有设定保持一致。

设计决策追踪是另一个重要应用。每一次设计评审会议的结论、每一条被采纳的玩家反馈、每一个版本的改动说明都可以通过 Graphiti 的 LLM 管线自动提取关键决策信息并纳入知识图谱。当团队成员对某个设计细节产生分歧时，可以查询知识图谱了解这个设计最初的理由和经历过的讨论，减少重复争论和信息不对称。

叙事一致性维护在大型游戏项目中尤为关键。随着游戏世界的规模扩大和参与撰写的人员增多，角色性格的前后矛盾、剧情时间线的逻辑错误、物品设定的相互冲突等问题很容易出现。Graphiti 的知识图谱可以作为一个"世界观守护者"，在新内容被录入时自动检查是否与已有知识产生冲突，及时提醒创作者修正。

## Claw Cluster 集成展望

Claw Cluster（AI 员工集群）将以 Graphiti 作为构建和维护游戏世界共享认知的核心基础设施。作为 EchoThink 的智能核心，Claw Cluster 中的每一个 AI Agent 在执行任务时都将与知识图谱进行双向交互——既从图谱中查询所需的上下文信息，也将自己工作过程中产生的新知识写回图谱。例如，当一个负责角色设计的 AI Agent 创建了一个新 NPC 时，它会同时在知识图谱中记录这个 NPC 与已有角色的关系、所处的地理位置、携带的物品以及相关的任务信息。这些新增的知识立即对其他 Agent 可见，使得负责关卡设计的 Agent 可以即时将新 NPC 纳入关卡编排的考量中。

这种基于知识图谱的协作模式使得 AI 员工集群具备了真正的"集体智慧"——每个 Agent 的工作成果都会丰富整个团队的知识积累，而每个 Agent 的决策都建立在整个团队的知识基础之上。随着项目的推进和知识图谱的不断丰富，AI Agent 的决策质量将持续提升，形成正向的知识增长飞轮。

## 关键配置参考

Graphiti Server HTTP API 端口：8000
Graphiti 健康检查端点：`http://localhost:8000/healthcheck`
Neo4j Bolt 协议端口：7687
Neo4j HTTP 接口端口：7474
Neo4j 内存限制：4GB（堆最大 512MB，页面缓存 256MB）
LLM 模型：`cost-effective`（通过 LiteLLM 路由）
嵌入模型：`embedding`（通过 LiteLLM 路由）
数据卷：`neo4j-data`
