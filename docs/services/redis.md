# Redis — EchoThink 的缓存与消息中枢

## 概述

Redis 在 EchoThink 基础设施中扮演着缓存层与消息代理的双重角色。我们部署的是 `redis:8.6.1-alpine` 镜像，选择 Alpine 变体是为了最小化容器体积和攻击面，同时保持 Redis 的全部功能。与 PostgreSQL 承担持久化数据存储不同，Redis 在 EchoThink 中被明确定位为纯粹的易失性缓存和消息传递层——持久化功能被完全禁用，所有数据仅存在于内存中。这一设计决策反映了 EchoThink 对"每个组件专注于自己最擅长的事情"这一原则的坚持：持久化的职责交给 PostgreSQL 和 MinIO，而 Redis 专注于提供亚毫秒级的读写速度和高效的发布/订阅消息传递。

Redis 容器通过挂载自定义配置文件 `/usr/local/etc/redis/redis.conf` 启动，该配置文件覆盖了 Redis 的默认行为，针对 EchoThink 的多服务共享场景进行了专门调优。容器命名为 `echothink-redis`，仅加入 `echothink-internal` 内部网络，不对外暴露任何端口，确保只有同一 Docker 网络内的服务才能与之通信。健康检查使用 `redis-cli ping` 命令，每 10 秒检测一次连通性，超时 5 秒，最多重试 5 次。

## 内存管理与淘汰策略

Redis 的内存配置是其作为共享缓存层最关键的设计决策之一。EchoThink 将 `maxmemory` 设置为 512MB，这个数字是经过仔细权衡的——它足以容纳所有服务的热点缓存数据，同时不会对宿主机的内存预算造成过大压力，将更多内存留给 PostgreSQL 的 shared_buffers 和各服务容器自身的运行需求。内存淘汰策略选择了 `allkeys-lru`，这意味着当内存使用达到上限时，Redis 会从所有键（而非仅设置了过期时间的键）中淘汰最近最少使用的数据。这种策略特别适合 EchoThink 的使用场景——多个服务的缓存数据混合存储在同一个 Redis 实例中，各服务的缓存访问模式差异较大，`allkeys-lru` 策略能够自动地将内存分配给最活跃的服务，而无需人工为每个服务预分配固定的内存配额。

持久化方面，RDB 快照和 AOF 日志均被显式禁用（`save ""` 和 `appendonly no`）。这意味着如果 Redis 容器重启，所有缓存数据将丢失。这在设计上是完全可接受的，因为缓存数据本质上是可重建的——各服务在缓存未命中时会回落到 PostgreSQL 查询并重新填充缓存。禁用持久化还带来了性能上的好处：Redis 不需要执行任何后台的磁盘写入操作，所有 CPU 周期都用于处理客户端请求，降低了延迟的波动性。

## 多服务共享的使用模式

EchoThink 中有七个服务依赖 Redis，每个服务以不同的方式使用这个共享的缓存层。

Dify 是 Redis 最重要的消费者之一。Dify 使用 Redis 作为 Celery 的消息代理（message broker），所有的异步任务——包括 AI 模型调用、文档解析、知识库索引构建等——都通过 Redis 的列表数据结构进行排队和分发。Celery worker 从 Redis 中消费任务消息并执行处理，处理结果也暂存在 Redis 中等待调用方轮询。这种基于 Redis 的任务队列模式使 Dify 能够将耗时的 AI 推理操作异步化，前端用户界面保持响应，同时多个 worker 可以并行处理不同的任务。


n8n 利用 Redis 缓存工作流执行过程中的中间状态和节点输出数据。当复杂的自动化工作流包含多个步骤时，前序节点的输出需要暂存以供后续节点使用，Redis 的快速读写能力确保了工作流执行的流畅性。

LiteLLM 使用 Redis 缓存 AI 模型的推理响应。当相同或极其相似的请求在短时间内重复出现时，LiteLLM 可以直接从 Redis 缓存中返回结果，避免重复调用昂贵的大语言模型 API。这对于游戏开发中的批量内容生成场景尤为有价值——例如为一批类似的 NPC 生成对话模板时，很多请求的核心提示词是相同的，缓存命中可以显著降低 API 调用成本和响应延迟。

Outline、Langfuse 和 Hatchet 也各自使用 Redis 进行会话管理和临时数据缓存。Outline 缓存用户的编辑会话状态以支持实时协作编辑，Langfuse 缓存追踪数据的聚合结果以加速仪表盘的渲染，Hatchet 使用 Redis 协调分布式任务引擎中的工作者节点注册和心跳检测。

## 连接与性能配置

Redis 的网络配置面向容器内部通信进行了优化。监听地址绑定到 `0.0.0.0`，端口为标准的 6379。保护模式被关闭（`protected-mode no`），因为 Redis 运行在 Docker 内部网络中，网络隔离已经由 Docker 的 `echothink-internal` 网络提供，不需要 Redis 层面的额外访问控制。TCP backlog 设置为 511，tcp-keepalive 为 300 秒，确保长连接的稳定性。最大客户端连接数设置为 10000，这远超当前需求，但为未来服务扩展预留了充足空间。

慢日志阈值设置为 10000 微秒（10 毫秒），任何执行时间超过此阈值的命令都会被记录到慢日志中，最多保留 128 条记录。延迟监控阈值设置为 100 微秒，Redis 会持续追踪各类操作的延迟分布，为性能诊断提供数据。Lua 脚本的执行超时设置为 5000 毫秒，防止复杂的 Lua 脚本阻塞 Redis 的单线程事件循环。

## 游戏开发中的应用价值

在游戏设计与开发的日常工作中，Redis 的价值体现在多个层面。首先是 AI 推理结果的缓存。当团队使用 Dify 构建游戏剧情生成器或 NPC 对话系统时，Redis 缓存了热门角色模板的推理结果，使得对相同角色设定的重复查询能够在毫秒级返回，而不是每次都等待数十秒的模型推理。这对于需要快速迭代的游戏设计流程至关重要——设计师可以反复微调提示词并几乎即时看到结果，因为基础模板的推理结果已经被缓存。

其次是实时协作状态的管理。当多位设计师同时在 Outline 中编辑游戏设计文档时，Redis 作为中间状态层确保每个用户都能看到其他协作者的最新修改。n8n 自动化工作流中的游戏资产处理管线——例如接收 AI 生成的概念图后自动触发格式转换、元数据标注和团队通知——也依赖 Redis 的消息传递能力在各个处理步骤之间高效传递数据。

此外，Hatchet 任务引擎通过 Redis 协调多个 worker 节点的任务分配，这对于批量的游戏资产生成任务（如一次性为数十个游戏角色生成立绘和背景故事）的并行处理至关重要。Redis 的发布/订阅功能使得新任务的分发延迟降到最低，所有空闲的 worker 都能立即感知并开始处理新任务。

## Claw Cluster 集成展望

Claw Cluster（AI 员工集群）上线后，Redis 将成为 AI 代理之间协调与通信的核心基础设施。Claw Cluster 中的多个 AI 代理需要共享工作状态、分配子任务、传递中间结果，Redis 的列表和发布/订阅机制天然适合这种多代理协作模式。例如，一个负责游戏关卡设计的 AI 代理可以通过 Redis 将"需要为第三关创建三种敌人类型"的子任务发布出去，由专门负责角色设计的 AI 代理接收并处理，处理完成后的角色数据再通过 Redis 回传给关卡设计代理进行整合。

Claw Cluster 还将大量使用 LiteLLM 的 Redis 缓存功能来降低 AI 推理成本。当多个 AI 代理处理相似的游戏设计任务时，它们发出的 LLM 请求往往具有较高的重复率——例如查询同一个游戏世界观的背景设定，或者使用相同的系统提示词。Redis 缓存层将自动识别这些重复请求并直接返回缓存结果，预计可以将 Claw Cluster 的 API 调用成本降低 30% 以上。

## 关键配置速查

- 镜像：`redis:8.6.1-alpine`
- 容器名称：`echothink-redis`
- 端口：6379（仅内部网络可访问）
- 最大内存：512MB
- 淘汰策略：allkeys-lru
- 持久化：禁用（纯缓存模式）
- 最大客户端数：10000
- 慢日志阈值：10ms
- 网络：`echothink-internal`
- 数据卷：`redis-data`
