# n8n — EchoThink 的自动化工作流引擎

## 服务概述

n8n 是 EchoThink 基础设施中的自动化工作流引擎，当前部署版本为 2.12.0。在整个平台架构中，n8n 扮演着"神经系统"的角色——它将散布在不同服务中的能力串联起来，让原本孤立的工具之间能够以事件驱动的方式协同工作。对于一个专注于游戏设计与开发的团队而言，日常工作中充斥着大量重复性操作：美术资源提交后需要触发构建管线、代码合并后需要通知相关设计师、测试报告生成后需要自动归档到知识库。这些琐碎但不可或缺的环节，正是 n8n 大显身手的场景。

n8n 提供了一个直观的可视化工作流构建器，团队成员无需编写复杂的脚本就能通过拖拽节点的方式搭建自动化流程。平台内置了超过四百种集成连接器，覆盖了从消息通知、数据库操作到第三方 API 调用的各类需求。更重要的是，n8n 的 HTTP Request 节点让它具备了调用任意 API 的能力，这意味着 EchoThink 平台内的所有服务——Dify 的 AI 对话接口、Outline 的文档管理 API、GitLab 的代码仓库接口——都可以被 n8n 的工作流直接调用和编排。这种灵活性使得团队可以根据自身的游戏开发流程定制任意复杂度的自动化方案，而不受预设模板的局限。

## 部署架构与配置

在 EchoThink 的部署方案中，n8n 以 Docker 容器形式运行，使用官方镜像 `docker.n8n.io/n8nio/n8n:2.12.0`，限定在 `linux/amd64` 平台上。容器内部服务监听 5678 端口，该端口映射到宿主机的 `127.0.0.1:5678`，仅对本机可达，所有外部访问均通过 Nginx 反向代理以 HTTPS 方式提供服务。Nginx 配置中定义了 `n8n-backend` 上游，将请求转发至容器内的 5678 端口，并特别配置了 WebSocket 升级支持（通过 `Upgrade` 和 `Connection` 头部），因为 n8n 的编辑器界面依赖 WebSocket 进行实时通信。代理读取超时设置为 300 秒，以适应长时间运行的工作流执行场景，同时关闭了代理缓冲和分块传输编码，确保工作流执行结果能够实时回传。

n8n 的数据持久化依赖 EchoThink 共享的 PostgreSQL 16 实例。工作流定义、凭证信息、执行历史等核心数据全部存储在名为 `n8n` 的专用数据库中，由用户 `n8n_user` 进行访问。同时，n8n 依赖共享的 Redis 实例进行缓存操作，这在高频触发的工作流场景中能够显著降低数据库压力。容器启动时会等待 PostgreSQL 和 Redis 均通过健康检查后才开始初始化，确保数据层就绪。服务的本地数据卷 `n8n-data` 挂载到容器内的 `/home/node/.n8n` 目录，用于存储运行时配置和临时文件。

关键环境变量与端口信息如下：

- 内部端口：5678
- 宿主机映射：127.0.0.1:5678
- 外部访问：https://n8n.${DOMAIN}（经 Nginx 反向代理）
- `N8N_ENCRYPTION_KEY`：用于加密存储的凭证信息
- `N8N_DB_PASSWORD`：PostgreSQL 数据库连接密码
- `WEBHOOK_URL`：Webhook 回调的公开 URL

## 工作流触发与执行机制

n8n 的核心能力之一是基于 Webhook 的事件驱动触发机制。每个工作流都可以配置一个独立的 Webhook 端点，当外部系统向该端点发送 HTTP 请求时，工作流会被立即触发执行。这个机制在游戏开发场景中有着丰富的应用：GitLab 的 Webhook 可以在代码推送、合并请求创建或 CI/CD 管线状态变更时通知 n8n，由 n8n 工作流决定后续操作——是通知相关团队成员，还是触发自动化测试，或是更新项目管理面板上的状态。Webhook URL 被配置为 `https://n8n.${DOMAIN}`，确保所有回调请求都通过 TLS 加密传输。

在执行管理方面，n8n 被配置为 `regular` 执行模式，适合单实例部署场景。系统会完整保存所有执行记录，无论成功还是失败，并且支持执行过程中的进度保存和手动触发的执行记录保存。执行数据的自动清理功能已开启，保留周期设为 336 小时（即 14 天），这为团队提供了充足的执行历史回溯窗口，便于调试和审计自动化流程的运行状况。当某个游戏构建管线的自动通知流程出现异常时，开发者可以方便地在 n8n 的执行历史中追溯具体哪一步出了问题，查看每个节点的输入输出数据，快速定位故障原因。

## 凭证安全与集成管理

安全性是自动化平台不可忽视的维度。n8n 通过 `N8N_ENCRYPTION_KEY` 对所有存储的凭证信息进行加密，确保 API 密钥、数据库密码和 OAuth 令牌等敏感数据在持久化存储中始终处于加密状态。这意味着即使数据库被未授权访问，攻击者也无法直接获取明文凭证。团队在 n8n 中配置的所有集成凭证——连接 Dify 的 API Key、访问 GitLab 的 Personal Access Token、操作 MinIO 的存储密钥——都受到这一层加密保护。在实际使用中，建议为每个集成服务创建专用的 API 凭证，遵循最小权限原则，避免使用管理员级别的令牌。

n8n 的遥测功能和版本更新通知在部署中已被关闭（`N8N_DIAGNOSTICS_ENABLED` 和 `N8N_VERSION_NOTIFICATIONS_ENABLED` 均设为 false），这符合 EchoThink 自托管基础设施对隐私和可控性的要求。所有数据流转和工作流逻辑都完全在团队自己的基础设施内完成，不存在向外部发送数据的风险。

## 游戏开发场景中的应用价值

对于游戏开发团队来说，n8n 的自动化能力可以深入到日常工作的方方面面。在美术资源管线中，当美术设计师将新的角色模型或场景贴图提交到 GitLab 仓库时，n8n 可以自动触发资源处理流程——调用构建服务进行格式转换和压缩、将处理结果上传到 MinIO 存储桶、在 Outline 知识库中更新资源清单文档，并向相关的技术美术和程序员发送通知。整个过程无需人工干预，显著减少了协作中的等待时间和沟通成本。

在构建状态监控方面，n8n 可以定期轮询 GitLab 的 CI/CD 管线状态，当检测到构建失败时自动收集错误日志、关联最近的代码变更记录，并生成结构化的故障报告推送给责任人。在文档自动更新场景中，当 API 接口发生变更时，n8n 可以自动调用 Dify 的 AI 能力生成更新后的接口文档，再通过 Outline 的 API 将新文档发布到团队知识库中。这些自动化流程把原本需要多人协调完成的工作压缩成了秒级完成的自动化链路。

## 与 Claw Cluster 的协同展望

即将上线的 Claw Cluster（AI 员工集群）是 EchoThink 平台的智能核心，而 n8n 将成为 Claw Cluster 与整个基础设施之间的关键桥梁。Claw Cluster 中的 AI 智能体将能够通过 n8n 的 Webhook 端点以编程方式触发工作流，实现跨服务的复杂操作编排。设想这样一个场景：一个负责游戏关卡设计的 AI 智能体在 Dify 中完成了关卡方案的生成，它会触发 n8n 工作流将方案文档自动写入 Outline 知识库、在 GitLab 中创建对应的开发任务 Issue、通知相关的程序员和美术设计师进行评审，并在 Hatchet 任务引擎中注册后续的跟踪任务。整个流程由 n8n 工作流统一编排，AI 智能体只需发出一个触发信号，后续的多服务协调操作便自动完成。

反过来，n8n 也将成为 Claw Cluster 感知外部事件的通道。当 GitLab 收到外部合作伙伴的代码贡献、当 Outline 知识库中的设计文档被更新、当 CI/CD 管线的构建结果发生变化时，n8n 工作流可以将这些事件信息格式化后推送给 Claw Cluster 中的相应 AI 智能体，让它们根据最新的项目状态调整自己的工作计划。通过 n8n 这个自动化中枢，Claw Cluster 中的每一个 AI 智能体都不再是孤立运行的个体，而是能够感知环境变化、协调多方资源、驱动端到端流程的智能协作者。这种人机协同的工作模式将为游戏开发带来前所未有的效率提升。
