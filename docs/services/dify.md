# Dify — EchoThink 的 AI 应用开发平台

## 平台定位与核心价值

Dify 1.13.0 是 EchoThink 基础设施中的 AI 应用开发平台，它为团队提供了一个可视化的工作流构建环境，使得无需编写复杂代码即可创建、调试和部署 AI 应用。对于我们这支专注于游戏设计与开发的团队而言，Dify 的意义远不止于一个简单的 AI 调用界面。它是连接大语言模型能力与实际游戏开发需求之间的桥梁，让策划人员、美术设计师和程序员都能够以直观的方式构建属于自己的 AI 辅助工作流。无论是生成 NPC 对话、自动撰写关卡描述文案，还是构建一套完整的美术资产标注流水线，Dify 都能以拖拽节点、连接逻辑的方式快速实现原型。

在 EchoThink 的整体架构中，Dify 通过 LiteLLM 网关统一访问各种大语言模型，这意味着团队在 Dify 中创建的任何应用都不需要关心底层使用的是哪个模型供应商。配置文件中的 `OPENAI_API_BASE` 指向 `http://litellm:4000/v1`，所有模型请求都经由 LiteLLM 路由和负载均衡，团队可以在 LiteLLM 侧自由切换模型而无需修改 Dify 中的任何应用配置。

## 容器架构与服务组成

Dify 在 EchoThink 中由四个容器协同工作，各自承担不同的职责。API 服务容器（dify-api）是整个平台的核心，基于 Flask 框架构建，运行在 5001 端口上，负责处理所有的 API 请求、管理应用逻辑、执行工作流编排以及与外部服务的通信。Web 前端容器（dify-web）运行在 3000 端口，提供基于 Next.js 构建的可视化管理界面，包括工作流编辑器、应用管理面板和对话测试窗口。Worker 容器（dify-worker）使用与 API 相同的镜像，但以 Celery worker 模式运行，通过 gevent 并发处理四个工作协程，负责处理 dataset（知识库索引）、generation（内容生成）、mail（邮件通知）和 ops_trace（运维追踪）四个异步任务队列。Sandbox 容器（dify-sandbox，版本 0.2.12）提供安全的代码执行沙箱环境，运行在 8194 端口，工作超时设置为 15 秒，允许网络访问但严格限制执行范围。

这四个容器之间的依赖关系经过精心编排：Sandbox 首先启动，API 服务依赖 Sandbox 就绪后启动，Web 前端依赖 API 服务可用后启动，Worker 则依赖 API 和 Sandbox 都准备完毕后才开始消费任务队列。API 和 Web 同时连接到 echothink-internal 和 echothink-public 两个网络，前者用于与其他内部服务通信，后者通过 Nginx 反向代理对外提供访问。Worker 和 Sandbox 仅连接内部网络，确保异步任务处理和代码执行不会直接暴露在外。

## 可视化工作流与 Agent 模式

Dify 最具特色的能力在于其可视化工作流构建器。团队成员可以在图形界面中以节点方式组合 LLM 调用、条件判断、代码执行、HTTP 请求、知识库检索等操作，构建出复杂的 AI 处理流水线。工作流配置中设定了最大执行步骤数为 500 步（`WORKFLOW_MAX_EXECUTION_STEPS`），最大执行时间为 1200 秒（`WORKFLOW_MAX_EXECUTION_TIME`），嵌套调用深度限制为 5 层（`WORKFLOW_CALL_MAX_DEPTH`），这些参数确保了工作流既有足够的灵活性来处理复杂场景，又不会因为意外的无限循环而耗尽系统资源。

Agent 模式赋予了 Dify 中的 AI 应用工具调用能力。一个配置了 Agent 模式的应用可以根据用户的输入自主决定调用哪些工具来完成任务，比如先查询知识库获取游戏世界观设定，再调用代码沙箱计算数值平衡，最后生成一份完整的技能设计文档。这种自主决策的能力使得 AI 应用从简单的问答系统升级为真正的智能助手。

## RAG 知识库与文件存储

Dify 内建了完整的 RAG（检索增强生成）系统，支持将各类文档上传为知识库，供 AI 应用在回答问题或执行任务时检索参考。当前配置使用 Weaviate 作为向量存储后端（`VECTOR_STORE=weaviate`），ETL 处理引擎设置为 Dify 内置引擎（`ETL_TYPE=dify`），索引时每个文档片段的最大 token 长度为 1000。文件上传限制为单文件 15MB、单次批量上传 5 个文件、图片文件最大 10MB，这些限制在 `config/.env` 中可以按需调整。

所有文件存储通过 S3 协议对接 MinIO 对象存储服务，使用 `dify-storage` 存储桶，配置了 path-style 访问模式。这种设计确保了 Dify 的文件存储与 EchoThink 整体的对象存储基础设施统一管理，团队可以在 MinIO 控制台中统一查看和管理所有服务的存储使用情况。

## 代码执行沙箱

Sandbox 容器为 Dify 提供了安全的代码执行环境，这对于游戏开发场景尤为重要。当 AI 需要生成并验证代码片段时——比如生成一段伤害计算公式、编写一个简单的地图生成算法、或者处理数据格式转换——代码会被发送到 Sandbox 中隔离执行，执行结果返回给工作流的下一个节点。Sandbox 对代码执行施加了严格限制：数字范围限制在正负 9999999999999 之间，字符串最大长度 80000 字符，字符串数组和对象数组最大 30 个元素，数字数组最大 1000 个元素。这些限制既保证了大多数游戏开发场景的需求，又防止了恶意或错误代码对系统造成伤害。

## 游戏开发场景中的应用

在游戏开发的日常工作中，Dify 可以发挥多方面的作用。策划团队可以构建对话生成工作流，输入角色性格设定和场景描述，批量生成符合风格的 NPC 对话文本。关卡设计师可以创建一个关卡建议应用，输入难度曲线参数和玩家行为数据，让 AI 提出关卡调整建议。美术团队可以搭建资产描述生成流水线，将概念图或模型截图上传，自动生成符合项目规范的资产命名和描述。程序团队可以利用 Agent 模式创建代码审查助手，结合项目代码规范知识库，对提交的代码片段进行风格检查和优化建议。

## Claw Cluster 集成展望

即将上线的 Claw Cluster（AI 员工集群）将以 Dify 作为构建和运行 AI 工作流的主要界面。Claw Cluster 作为 EchoThink 的智能核心，其中每一种类型的 AI Agent 都将拥有自己独立的 Dify 应用——对话生成 Agent 对应一个精调了叙事技巧的工作流，数值平衡 Agent 对应一个集成了数据分析工具链的工作流，美术资产 Agent 对应一个包含视觉理解和描述生成能力的工作流。Dify 的工作流可视化特性使得团队可以直观地理解每个 AI Agent 的决策逻辑，随时调整其行为参数和工具配置，而无需深入到代码层面。这种透明性对于确保 AI 员工的输出质量与团队预期一致至关重要。

## 关键配置参考

Dify API 健康检查端点：`http://localhost:5001/health`
Dify Web 前端端口：3000
Dify Sandbox 端口：8194
Celery 消息队列：Redis DB 1（`redis://redis:6379/1`）
应用数据 Redis：Redis DB 2
数据库：PostgreSQL `dify` 库，用户 `dify_user`
对象存储：MinIO `dify-storage` 桶
模型网关：`http://litellm:4000/v1`
