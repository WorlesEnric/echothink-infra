# Open-Source Stack Architecture for Human-AI Real-Time Collaboration

## Chapter 1: Introduction and Motivation

This document describes a complete open-source technology stack designed to power a real-time human-AI collaboration system in which a small entrepreneurial team works alongside a massive fleet of AI agents as employees. The system is not a chatbot wrapper or a simple automation pipeline. It is a structured working environment where human team members and AI agents share project state, coordinate through a common object and relationship layer, and participate in the same governance and approval processes. The AI agents are treated as first-class employees — they have identities, permissions, execution environments, assigned tasks, observable outputs, and accountability records just like their human counterparts.

The original prototype was built as a monorepo with custom services for identity, studio collaboration, console governance, agent orchestration, event distribution, knowledge retrieval, and observability. While the prototype successfully validated the design, the team recognized that the reliability, scalability, and maintenance burden of a fully custom stack would be difficult to sustain. The decision was made to replace the custom services with a carefully composed set of mature open-source projects, each chosen because it excels at a specific layer of the system and integrates cleanly with the others.

The stack described here is not a theoretical exercise. Every project mentioned has been evaluated for self-hosting capability, licensing compatibility, active maintenance, and production readiness. The goal is a system where the team operates its own infrastructure, retains full control over data and governance, and avoids vendor lock-in while benefiting from the combined engineering effort of thousands of open-source contributors.

## Chapter 2: The Collaboration Model

Before examining the technology, it is important to understand the collaboration model that the stack must support. The system serves a team of roughly a dozen human members who are building software products — applications, games, and algorithm services. Alongside them, a potentially large number of AI agents operate as employees. These agents are not passive tools that respond only when prompted. They are active participants that monitor project state, propose changes, generate artifacts, execute assigned work, and report results back into the shared workspace.

The collaboration happens in workspaces, each corresponding to a project or working context. Within a workspace, both humans and agents interact with panels — independent service modules that provide specific capabilities such as task management, note editing, flowchart design, code review, or report generation. Each panel is a self-contained application with its own frontend and backend, but all panels share a common coordination layer managed by the studio. This coordination layer maintains an object repository where panels register addressable entities, and a link graph where relationships between objects from different panels are recorded. The result is a shared map of project meaning that any participant — human or AI — can navigate, query, and update.

The agents work through structured operations rather than free-form interactions. An agent enters a workspace with a defined identity, a bounded set of permissions, access to selected skills, and a specific execution environment. It reads workspace state by inspecting registered objects and traversing the link graph. It proposes changes through formal channels that may require human approval. It executes approved work in sandboxed environments. It writes results back as new objects, links, status updates, and artifacts. Every action is recorded in an event stream and an audit log.

This model demands infrastructure that can handle real-time bidirectional communication between many concurrent participants, durable execution of long-running agent workflows, temporal awareness of how system state has changed over time, fine-grained access control, and comprehensive observability. The open-source stack described below addresses each of these demands with a dedicated, proven component.

It is worth placing this model in context. The idea of AI agents as formal team members is no longer speculative. Gartner forecasts that by 2028, thirty-eight percent of organizations will have AI agents listed as formal team members within human teams, with defined roles, responsibilities, and performance metrics. Microsoft has declared 2026 the year of the agent, envisioning workflows where a three-person team launches a global campaign in days with AI handling data analysis, content generation, and personalization while humans steer strategy. McKinsey frames this as "skill partnerships" — humans and agents each contributing what they do best. At the same time, IDC's Future of Work research pushes back on the "coworker" framing, arguing that AI systems are instruments, not peers — programmable, bounded, and entirely dependent on human judgment. This tension is precisely why the collaboration system needs strong governance infrastructure. The agents are powerful participants, but they operate within boundaries set and enforced by humans. The stack must make those boundaries explicit, observable, and non-negotiable.

## Chapter 3: PostgreSQL as the Converged Data Foundation

At the center of the entire stack sits PostgreSQL, the relational database that has become the dominant data platform for AI-era applications. The choice of PostgreSQL as the primary data store is not merely a preference for familiarity — it reflects a deep industry trend. In 2025, Databricks acquired Neon, the serverless PostgreSQL company, for one billion dollars. Snowflake acquired Crunchy Data, the PostgreSQL support and tooling leader, for two hundred and fifty million dollars. Supabase, the open-source backend-as-a-service built entirely on PostgreSQL, reached a five billion dollar valuation. These acquisitions and valuations signal that the industry has converged on PostgreSQL as the universal data layer for modern applications.

PostgreSQL serves as a converged database in this stack, meaning it handles relational data, document storage through its JSONB column type, full-text search through tsvector and pg_trgm, and vector similarity search through the pgvector extension. The advantage of convergence is operational simplicity. Instead of running separate databases for relational state, document queries, keyword search, and embedding retrieval, the team manages a single PostgreSQL cluster where all these capabilities coexist within the same transactional boundary. Queries can join relational tables with vector similarity results in a single SQL statement. Row-level security policies protect all data types uniformly. Backup and replication cover everything at once.

The pgvector extension deserves particular attention because it provides the vector search capability that powers the knowledge retrieval layer of the collaboration system. pgvector supports three index strategies: IVFFlat, which partitions vectors into clusters and searches the most relevant clusters during query time; HNSW, which builds a hierarchical navigable small world graph for approximate nearest neighbor search; and flat scanning with no index, which provides exact results at the cost of linear scan time. HNSW generally provides better recall and more consistent query latency, handles dynamic data without requiring index rebuilds, and is the recommended default for most workloads. Version 0.7.0 introduced parallel HNSW index builds that are up to thirty times faster than serial builds, and combined with the halfvec compression type that reduces index size by fifty percent with minimal recall impact, index creation is now up to sixty-seven times faster than earlier versions on cloud-managed PostgreSQL instances.

pgvector handles up to approximately five to ten million vectors on a single PostgreSQL instance with HNSW, with query latencies in the low tens of milliseconds. This is more than sufficient for the knowledge base of any project workspace in this system. The practical advantage of pgvector over dedicated vector databases is co-location with relational data — a single SQL query can join a vector similarity search with relational filters, workspace membership checks, and Row Level Security policies, all within one transaction. The disadvantage is that index builds are memory-intensive and compete with production query workload, and there is no native sharding for vector data across multiple PostgreSQL instances. For workloads where vector operations dominate or data exceeds tens of millions of vectors, a dedicated vector database like Qdrant or Weaviate should be introduced. But for the collaboration system's scale, pgvector within PostgreSQL eliminates the need for a separate vector store while providing the transactional consistency and access control that no standalone vector database can match.

PostgreSQL also provides the temporal modeling foundation through tstzrange types, system-versioned temporal tables, and the temporal_tables extension. These features allow the system to record not just the current state of objects and relationships, but the full history of when each state was valid and when it was superseded. This temporal capability is essential for the knowledge graph layer described later.

## Chapter 4: Supabase as the Application Data Platform

While PostgreSQL provides the raw database engine, Supabase wraps it with a comprehensive application platform that eliminates enormous amounts of boilerplate that the prototype originally implemented by hand. Self-hosted Supabase is a composition of approximately twelve interconnected open-source services that work together as a cohesive whole. At its core is a standard, unforked PostgreSQL instance, unchanged and fully accessible through standard SQL connections. On top of this, Supabase adds PostgREST (written in Haskell), which automatically generates a RESTful API from the database schema, and pg_graphql, which provides a GraphQL endpoint derived from the same schema. These auto-generated APIs mean that many of the custom route handlers in the prototype's backend services become unnecessary — the database schema itself defines the API contract. The stack also includes Supavisor (an Elixir-based connection pooler that replaced PgBouncer), Kong as the API gateway handling JWT validation and request routing, pg_meta for PostgreSQL metadata access, Edge Functions powered by Deno for serverless compute, and imgproxy for on-the-fly image transformation. All stateless components — PostgREST, Auth, Realtime, Storage — can be horizontally scaled behind a load balancer independently of each other.

Supabase Auth provides the authentication layer through GoTrue (written in Go), supporting email and password login, magic links, phone verification, OAuth providers, and JWT token issuance. When a user or agent authenticates, Supabase issues a JWT that contains claims about the actor's identity and role. This token flows through every subsequent API request and is evaluated by PostgreSQL's Row Level Security policies. Row Level Security is a PostgreSQL-native feature that attaches boolean predicates to table rows, evaluated on every read and write operation (SELECT, INSERT, UPDATE, DELETE separately). Supabase provides helper functions — auth.uid() to extract the current user's ID from the JWT, auth.jwt() to access the full JWT claims, and auth.role() to check the session role — that make writing RLS policies intuitive. For example, a policy might state that a user can only read objects where the workspace_id matches one of their authorized workspaces, or that an agent can only update rows where its identity matches the assigned_agent_id column. These policies are defined in SQL, version-controlled alongside schema migrations, and enforced at the database level regardless of which API path — REST, GraphQL, or direct SQL — is used to access the data. This is a dramatic improvement over the prototype's application-level permission checks, because it is impossible to bypass Row Level Security through a forgotten middleware or a misconfigured route. The RLS policies extend uniformly to Supabase Storage for file access control, to Realtime for channel authorization, and to Edge Functions for serverless compute boundaries.

Supabase Realtime is the component that enables live collaboration. It is built with Elixir on the Phoenix Framework, using Phoenix.PubSub with the PG2 adapter for distributed process group coordination across multiple Realtime nodes. The Realtime server provides three distinct capabilities. The first is Broadcast — ephemeral client-to-client messaging through channels, where a message sent by a participant in one geographic region is delivered to participants connected to different Realtime nodes in other regions through Erlang's built-in distributed process routing. The second is Presence — an in-memory CRDT-backed key-value store that tracks which users and agents are currently connected to each workspace, with state replicated across all Realtime nodes for consistency. The third is Postgres Changes, which is the most architecturally significant capability — the Realtime server polls PostgreSQL's Write-Ahead Log through a logical replication slot, appends channel subscription identifiers (which are globally-unique Erlang process IDs) to each WAL record, and routes the changes to the correct WebSocket connections through the Erlang VM's native process routing infrastructure. This means that when an agent registers a new object in the workspace, updates a task status, or creates a link between two entities, the change is written to PostgreSQL and immediately broadcast to all participants — human and AI — who have subscribed to that workspace's channel, with no application-level fan-out code required. This replaces the custom WebSocket gateway that the prototype built with Fastify and manual client tracking. The prototype's WebSocket implementation required explicit heartbeat management, reconnection logic, and per-workspace client set maintenance. Supabase Realtime handles all of this automatically through the Erlang VM's lightweight process model, which was designed from the ground up for millions of concurrent connections in telecommunications systems.

Supabase Storage provides S3-compatible object storage backed by MinIO or a local filesystem, with access policies that integrate with the same Row Level Security system used for database tables. When agents generate artifacts — images, documents, code patches, analysis reports — these files are stored in Supabase Storage with access policies that respect workspace boundaries and agent permissions.

For self-hosted deployment, Supabase provides a Docker Compose configuration that brings up all components — PostgreSQL, PostgREST, GoTrue (the auth service), Realtime, Storage, Kong (the API gateway), and the Studio dashboard — as a single coordinated stack. The team retains full control over their data, can customize PostgreSQL extensions, and can scale individual components independently. The Studio dashboard gives non-technical team members a visual interface for browsing data, managing users, and monitoring real-time connections without needing to use the admin-web application directly.

## Chapter 5: Authentik as the Identity and Access Governance Layer

The collaboration system requires an identity provider that goes beyond simple authentication. It must manage human users, AI agent identities, service accounts, workspace memberships, role assignments, permission scopes, OAuth integrations with external services like GitLab, and protocol bridges for legacy systems. Authentik is the identity provider chosen for this role because it provides all of these capabilities in a single, self-hosted, open-source platform with a modern architecture.

Authentik is built as a Django application with a PostgreSQL backend, which means it shares the same database technology as the rest of the stack. Its core distinguishing feature is the concept of customizable authentication flows. Rather than offering a fixed login process, Authentik allows administrators to compose authentication flows from building blocks — stages — that can include password verification, multi-factor authentication, email confirmation, invitation codes, consent screens, and custom policy evaluations. This flexibility is critical for the collaboration system because human users and AI agents have fundamentally different authentication paths. A human user might authenticate through an OAuth flow with their GitLab account and then confirm a second factor. An AI agent might authenticate through a service account token issued by the console with a specific set of scopes that limit what it can access within a workspace.

Authentik's outpost architecture extends its reach beyond the core server. Outposts are standalone Go binaries compiled with FIPS 140 cryptographic modules, acting as protocol adapters that translate various authentication protocols into interactions with the Authentik backend API. The proxy outpost provides forward authentication for reverse proxies like Nginx, Traefik, or Caddy, listening on ports 9000 (HTTP) and 9443 (HTTPS), and enabling any web application to be protected by Authentik without modifying the application's code. It supports both single-application mode, where each application has its own access policies evaluated independently, and domain-level mode, where a single configuration covers all applications under a domain. The LDAP outpost translates LDAP bind and search operations into Authentik API calls, supporting legacy systems that require directory-based authentication. For geographically distributed deployments, the LDAP outpost supports local caching — if the team has infrastructure in multiple locations, the outpost can serve authentication requests from its cache without round-tripping to the central Authentik server, which is critical for latency-sensitive operations. The RADIUS outpost handles network authentication for VPN and WiFi access. All outposts fetch their configuration through the Authentik API and receive real-time configuration updates over WebSocket, meaning that policy changes propagate to all outposts without restarts. These outposts can be deployed anywhere and communicate with the central Authentik instance over its API, which means the collaboration system can extend authentication to any service in the infrastructure — panel backends, GitLab, internal tools — through a single identity provider.

Compared to Keycloak, the other major open-source identity provider, Authentik offers a significantly more intuitive administration interface, simpler deployment, lower memory footprint (it does not run on the JVM), and more flexible flow customization. Keycloak has a longer history and deeper integration with enterprise Java ecosystems, but for a small team running a modern Python and TypeScript stack, Authentik is the more natural fit.

The identity provider integrates with the rest of the stack through standard protocols. Supabase Auth can be configured to delegate to Authentik as an external OAuth provider. Hatchet's API authenticates agent workers through tokens issued by Authentik. Dify's user management can be federated through Authentik's OIDC provider. The result is a single source of truth for identity across the entire collaboration system.

## Chapter 6: Dify as the Agent Orchestration Engine

The collaboration system needs a way to define, execute, and manage the workflows that AI agents follow when performing their work. An agent's work is rarely a single LLM call — it is a multi-step process that might involve reading workspace state, querying the knowledge graph, generating artifacts, waiting for human review, processing feedback, and writing results back to the project. Dify is the platform chosen for this orchestration because it provides a visual workflow builder, a runtime execution engine, support for human-in-the-loop interactions, multi-provider LLM integration, and a plugin architecture for extending its capabilities — all in a self-hosted open-source package.

Dify's workflow engine is built on a Celery worker architecture with Redis as the message broker. When a workflow is triggered — either by a human request, a scheduled trigger, or an event from another part of the system — the API server dispatches the execution to a Celery queue named workflow_based_app_execution. A Celery worker picks up the task and begins executing the workflow graph node by node. During execution, streaming events are published back to the API server through Redis PubSub, enabling real-time progress updates. Starting with version 1.13, Dify supports sharded Redis PubSub and Redis Cluster mode, which are essential for scaling to the throughput required when hundreds of agents are executing workflows simultaneously.

The visual workflow builder allows the team to design agent behaviors as directed acyclic graphs. Each node in the graph represents an operation: an LLM call with a specific prompt template, a conditional branch based on the output of a previous node, an HTTP request to an external service, a code execution block for data transformation, a knowledge retrieval step, or a human input node that pauses the workflow and waits for a human decision. This last capability — the Human Input node introduced in version 1.13 — is what makes Dify suitable for the collaboration system's approval workflows. When an agent generates ten candidate images and needs a human to select one, the workflow pauses at the Human Input node, presents the options through the Dify interface or a connected frontend, and resumes execution only after the human makes a choice. The pause is durable — the workflow state is persisted in the database, so it survives server restarts and can remain pending for minutes, hours, or days.

Dify's plugin architecture, introduced in version 1.0, represents a fundamental shift in how the platform extends its capabilities. Models, tools, and agent strategies were migrated from hardcoded implementations to a plugin-based system where each capability is packaged as an independent module communicating with Dify through a defined interface. The most significant plugin category is Agent Strategies, which abstracts the decision-making logic that determines how an agent selects tools, plans actions, and reasons through multi-step problems. The built-in strategies include ReAct (interleaving reasoning and action), Function Calling (structured tool invocation), Chain-of-Thought (step-by-step reasoning), and Tree-of-Thought (exploring multiple reasoning branches). Because these strategies are plugins rather than hardcoded behaviors, the team can develop custom strategies optimized for their specific collaboration patterns — for instance, a strategy that always consults the temporal knowledge graph before proposing changes, or a strategy that explicitly decomposes workspace-level goals into panel-specific sub-tasks. The Agent Node in Dify's workflow builder serves as the decision-making center, dynamically calling models based on the selected strategy for resource scheduling, state management, inference recording, and tool selection. The collaboration system can provide additional tool plugins that connect Dify workflows to the studio's object repository, the link graph, the knowledge graph, and the event stream. When an agent workflow needs to register a new object in the workspace, it invokes a custom tool plugin that calls the studio's API. When it needs to traverse the temporal knowledge graph, it invokes a Graphiti plugin that queries FalkorDB. This plugin model keeps Dify's core clean while allowing arbitrarily deep integration with the rest of the stack.

Dify also provides multi-provider LLM integration out of the box. The team can configure access to OpenAI, Anthropic, Google Gemini, local models running through Ollama, and any other provider that exposes an OpenAI-compatible API. Different workflows can use different models depending on cost, capability, and latency requirements. A code review agent might use a fast, inexpensive model for initial analysis and escalate to a more capable model only when complex reasoning is needed. A creative content agent might route to a specialized image generation model for visual artifacts. The provider configuration is managed centrally, with API keys stored securely and usage tracked for cost observability.

For the collaboration system's scale — dozens of human team members and potentially hundreds of concurrent agent workflows — Dify's architecture is well-suited. The Celery worker pool can be scaled horizontally by adding more workers. The Redis PubSub layer can be scaled with Redis Cluster. The API server is stateless and can run behind a load balancer. The PostgreSQL backend benefits from the same scaling investments made for the rest of the stack.

## Chapter 7: Hatchet as the Durable Task Execution Engine

While Dify handles the high-level orchestration of agent workflows — the "what should the agent do" question — there is a complementary need for a lower-level task execution engine that handles the "how should individual tasks be reliably dispatched, retried, and coordinated" question. Hatchet fills this role as a distributed, fault-tolerant task queue built directly on PostgreSQL.

Hatchet's architecture is elegantly simple. It consists of three components: an engine that schedules and coordinates tasks, an API server that provides the management interface, and workers that execute individual task steps. All state is stored in PostgreSQL, which means the task execution system shares the same database infrastructure as the rest of the collaboration stack. There is no Redis, no RabbitMQ, no Kafka required for the core operation. Communication between the engine and workers happens through bidirectional gRPC connections that enable real-time task dispatch and status updates with minimal latency — sub-twenty-millisecond task start times are typical. The engine is designed to be horizontally scalable, with multiple engine instances coordinating through the shared PostgreSQL storage layer. Hatchet leverages PostgreSQL's FOR UPDATE SKIP LOCKED mechanism for queue management, which provides transactional guarantees that message brokers typically cannot offer. Workers which concurrently pull tasks off the queue never pull duplicate tasks because SKIP LOCKED prevents reading any rows locked by other worker transactions. A task is dequeued and processed within a database transaction — if the worker crashes, the transaction rolls back and the task becomes available for another worker. There is no message loss, no duplicate delivery under normal conditions, and no need for idempotency tokens.

Hatchet supports two workflow patterns that serve different needs of the collaboration system. The first is DAGs — directed acyclic graphs where the shape of work is known upfront, with declared tasks, ordering, and dependencies that Hatchet executes with automatic parallelism and retry logic. The second is Durable Tasks, a more flexible pattern where workflows can branch, loop, and spawn child tasks dynamically. In durable execution mode, the execution state is stored in Hatchet's event log, surviving worker crashes and resuming from the last checkpoint on any available worker. This durable execution model is essential for the collaboration system because agent workflows can span hours or days — waiting for human approval, monitoring external events, or coordinating across multiple workspace operations.

The scale that Hatchet has achieved on PostgreSQL is remarkable. From processing twenty thousand tasks in its first month of operation, Hatchet Cloud now processes over twenty thousand tasks per minute — more than one billion per month. Every task corresponds to at minimum five PostgreSQL transactions, meaning the engine sustains bursts of over twenty-five thousand transactions per second. For self-hosted deployments processing fewer than one hundred tasks per second, the pure PostgreSQL mode is sufficient. For higher throughput, Hatchet can optionally use RabbitMQ for inter-service communication while keeping PostgreSQL as the durable state store.

Hatchet's concurrency model is particularly well-suited for the collaboration system. Workers declare a number of slots representing their concurrent task capacity. If a worker is configured with five slots, it can execute five tasks simultaneously before new tasks queue. Beyond simple slot counting, Hatchet supports dynamic concurrency keys that enable multi-tenant fairness. The system can ensure that no single agent or user monopolizes execution resources — even if one agent has a thousand pending tasks, other agents' tasks will still be scheduled fairly. This prevents a runaway agent from starving the rest of the system.

Sticky worker assignment is another feature that matters for the collaboration system. When an agent workflow involves multiple steps that share expensive local state — perhaps a loaded machine learning model, a large file in memory, or an active database connection — all steps can be pinned to the same worker. Hatchet supports both soft sticky assignment, where the system prefers the same worker but falls back to another if unavailable, and hard sticky assignment, where the steps will only run on the original worker. Worker affinity labels extend this further — workers can declare capabilities like available GPU memory or loaded model versions, and the engine routes tasks to the most suitable worker based on weighted label matching.

In the collaboration system, Hatchet handles the execution layer beneath Dify. When a Dify workflow dispatches a task — "run this code analysis in a sandbox," "generate embeddings for this document," "sync this object to the knowledge graph" — the task is submitted to Hatchet, which manages queuing, worker assignment, retry logic, timeout handling, and result collection. Dify provides the workflow logic; Hatchet provides the reliable execution substrate.

## Chapter 8: The Temporal Knowledge Graph — Graphiti and FalkorDB

The collaboration system's most distinctive technical requirement is the temporal knowledge graph. Unlike a conventional database that stores the current state of the world, or a traditional knowledge graph that stores static facts and relationships, the temporal knowledge graph maintains a living, evolving record of system state with full awareness of time. When an agent needs to understand the current state of a project, it queries the graph for entities and relationships that are currently valid. When an agent needs to understand how the project reached its current state, it queries the graph for the temporal sequence of changes — what facts were established when, what facts were superseded, and why. This temporal dimension transforms the knowledge graph from a static reference into a genuine organizational memory.

Graphiti, developed by Zep, is the framework that provides the temporal knowledge graph semantics. The Zep architecture, described in an academic paper published in January 2025 (arXiv:2501.13956), organizes the knowledge graph into three hierarchical tiers of subgraphs. The first tier is the Episode subgraph, which stores raw conversational and event data — the unprocessed inputs that form the ground truth of everything the system has observed. The second tier is the Semantic Entity subgraph, which stores extracted entities and their typed relationships — the structured knowledge that agents actually query. The third tier is the Community subgraph, which stores higher-order clusters and patterns discovered across entities and relationships — emergent structures that help agents understand not just individual facts but broader themes and trends in the workspace.

At the data level, Graphiti implements a bi-temporal data model where every fact has two time dimensions: a valid_at timestamp recording when the fact became true in the real world, and an invalid_at timestamp recording when the fact was superseded by newer information. A third dimension, the ingestion timestamp, records when the system first learned about the fact. This three-dimensional temporal model enables point-in-time queries with remarkable precision — an agent can ask "what did the system know about deployment X at 3pm yesterday" and receive an accurate answer that reflects both the real-world state and the system's state of knowledge at that moment, even if both have changed multiple times since then. When new information contradicts a prior fact, Graphiti does not delete the old fact. Instead, it sets the old fact's invalid_at timestamp and creates a new fact with its own valid_at timestamp. This means the complete history of the system's understanding is preserved and queryable. On the Deep Memory Retrieval benchmark, the Zep architecture achieves 94.8 percent accuracy compared to MemGPT's 93.4 percent, and on the more challenging LongMemEval benchmark — which specifically tests cross-session information synthesis and long-term context maintenance — Zep achieves accuracy improvements of up to 18.5 percent while reducing response latency by ninety percent.

Graphiti's processing pipeline works through the concept of episodes. An episode is a unit of new information — a text message, a structured JSON event, an agent observation, a status update — that is submitted to the system for integration into the knowledge graph. When an episode arrives, Graphiti runs it through a multi-stage extraction pipeline powered by LLM calls. The first stage extracts entities — the people, services, concepts, and objects mentioned in the episode. The second stage extracts relationships — how those entities connect to each other based on the information in the episode. The third stage performs entity resolution, determining whether a newly extracted entity is the same as one already in the graph (for example, recognizing that "the studio service" and "studio-core" refer to the same thing). The fourth stage handles temporal invalidation, identifying which existing facts in the graph are contradicted by the new information and marking them as superseded. The fifth stage generates embeddings for both entities and relationships, enabling semantic similarity search alongside graph traversal. All of these stages run concurrently where possible, controlled by a configurable semaphore that limits the number of simultaneous LLM calls.

The result is a graph where entities are nodes and relationships are edges, each annotated with temporal metadata, semantic embeddings, and provenance information linking back to the original episodes. Querying this graph is not a simple keyword lookup — it is a hybrid retrieval process that combines semantic similarity (finding entities and relationships whose embeddings are close to the query embedding), keyword matching (BM25 scoring for precise term matches), and graph traversal (following relationship edges to discover connected context). The results from these three retrieval paths are merged using Reciprocal Rank Fusion, a technique that combines ranked lists from multiple sources into a single ranking that balances the strengths of each method.

FalkorDB is the graph database that stores and queries the knowledge graph. Chosen over Neo4j for its superior performance characteristics and more permissive self-hosting model, FalkorDB uses a fundamentally different computation engine built on SuiteSparse:GraphBLAS, Professor Tim Davis's implementation of the GraphBLAS standard for graph algorithms expressed as linear algebra operations on sparse matrices. While most graph databases traverse nodes and edges one at a time through pointer chasing, FalkorDB represents the entire graph as sparse adjacency matrices where a directed edge from node S to node T sets M[S,T]=1. Graph traversals are then translated into matrix operations. A friends-of-friends query becomes a matrix multiplication FOF = F * F. A complex pattern like (N0)-[A]->(N1)-[B]->(N2)<-[A]-(N3) is computed as A * B * Transpose(A). Because matrix multiplication is associative and distributive, the engine can choose the evaluation order that minimizes intermediate matrix density and maximizes parallelism, leveraging AVX SIMD instructions for hardware acceleration on modern CPUs. This approach transforms graph traversal from a sequential operation into massively parallel linear algebra that modern hardware executes extremely efficiently.

The performance difference compared to traditional graph databases is dramatic. FalkorDB's benchmarks show consistent sub-140-millisecond latency at the 99th percentile, while Neo4j experiences latencies exceeding forty seconds under comparable aggregate expansion workloads. At the 50th percentile, FalkorDB is roughly ten times faster, and the latency increase from p50 to p99 stays within a 2.5x factor, indicating stable and predictable performance under varying loads. These characteristics matter enormously for the collaboration system, where agents make knowledge graph queries as part of their decision-making loops and unpredictable latency spikes would cascade into degraded agent responsiveness. FalkorDB is written entirely in C with no JVM dependency, which means it uses far less memory than Neo4j, does not suffer from garbage collection pauses, and can run comfortably in a container with modest resource allocations. A next-generation Rust rewrite is also in progress, targeting even higher performance.

FalkorDB also provides a built-in vector index that integrates with its graph query engine. This means a single query can combine graph pattern matching with vector similarity search — for example, "find all entities within two hops of entity X whose embeddings are most similar to this query vector." This hybrid query capability is exactly what the temporal knowledge graph needs for RAG retrieval. Graphiti uses this capability through its pluggable driver architecture, which abstracts the underlying graph database behind a common interface. Switching between FalkorDB, Neo4j, or Kuzu requires no changes to application code.

For the collaboration system, the temporal knowledge graph serves as the shared memory that makes agents intelligent participants rather than stateless executors. When an agent begins working on a task, it queries the knowledge graph to understand the project context — what decisions have been made, what architecture has been established, what problems have been encountered, what previous work is relevant. Because the graph is temporal, the agent can distinguish between current facts and historical ones, understanding not just what is true now but what has changed and why. When the agent completes its work, the results are fed back into the knowledge graph as new episodes, which Graphiti processes and integrates, updating entities and relationships and invalidating any superseded information. Over time, the knowledge graph becomes an increasingly rich and accurate representation of the team's collective understanding.

The Graphiti MCP (Model Context Protocol) server provides a standardized interface that AI assistants and agents can use to interact with the knowledge graph directly. Through MCP, an agent can add episodes, search for relevant context, retrieve entity details, and traverse relationships without needing custom integration code. This aligns with the broader industry movement toward MCP as the standard protocol for connecting AI models to external knowledge and tools.

## Chapter 9: Redis as the Real-Time Communication and Caching Layer

Redis serves a focused but essential role in the stack as the real-time communication backbone, caching layer, and rate limiting engine. It is deliberately not used as a primary data store — that role belongs to PostgreSQL — but its in-memory, single-threaded architecture makes it uniquely suited for operations where latency matters more than durability.

The most critical use of Redis in the collaboration system is as the message broker for Dify's workflow execution engine. When a Celery worker executes an agent workflow, streaming events — progress updates, intermediate results, status changes — are published through Redis PubSub channels. The Dify API server subscribes to these channels and forwards events to connected clients in real time. For the collaboration system's scale, Redis Cluster with sharded PubSub ensures that this communication layer scales horizontally without becoming a bottleneck.

Redis also serves as the caching layer for frequently accessed data that is expensive to compute or retrieve. Panel configurations, workspace metadata, user session data, knowledge graph query results, and frequently accessed objects can all be cached in Redis with appropriate time-to-live settings. The caching strategy is straightforward: PostgreSQL is the source of truth, and Redis is a performance optimization that can be fully cleared and rebuilt at any time without data loss.

Rate limiting is the third major use of Redis. When many agents are operating simultaneously, the system needs to prevent any single agent from overwhelming shared resources — LLM provider APIs, external service endpoints, the knowledge graph ingestion pipeline. Redis supports multiple rate limiting algorithms suited to different aspects of the collaboration system. Token bucket rate limiting allows short bursts of activity while enforcing an average rate, which is ideal for the bursty nature of agent work — an agent might make several rapid LLM calls during an analysis phase and then remain quiet during a waiting phase. Sliding window rate limiting provides smoother enforcement for steady-state operations like knowledge graph ingestion. These rate limiters are implemented through Redis's atomic increment and expiration operations, with the checks executed as asynchronous middleware that does not block the main request thread — critical for compute-heavy LLM inference endpoints where blocking would cascade into system-wide latency degradation. The rate limits can be configured per-agent, per-provider, per-workspace, and globally, creating a layered defense against resource exhaustion.

Beyond these three primary roles, Redis also serves as the distributed locking mechanism for coordinating access to shared resources. When multiple agents need exclusive access to a workspace resource — for example, to prevent two agents from simultaneously modifying the same object — Redis distributed locks provide the coordination with minimal overhead. Redis Streams, the durable messaging feature, provides an alternative to PubSub for scenarios where message loss is unacceptable, such as critical event delivery between services.

## Chapter 10: Langfuse as the LLM Observability Platform

When dozens or hundreds of AI agents are executing workflows simultaneously, the team needs deep visibility into what those agents are doing, how well they are performing, what they are costing, and where they are failing. Langfuse is the open-source LLM observability platform that provides this visibility.

Langfuse traces every LLM interaction in the system, capturing the full lifecycle of each call: the prompt that was sent, the model that was used, the response that was returned, the latency of the call, the number of tokens consumed, and the estimated cost. But Langfuse goes beyond individual LLM calls to trace entire execution chains. When an agent workflow involves multiple LLM calls — an initial analysis, a knowledge graph query interpretation, a code generation step, and a summary — Langfuse traces all of these as a single session with parent-child relationships, showing how the calls connect and how each step influenced the next.

Langfuse integrates with the collaboration system at multiple points. Dify has a native Langfuse integration that automatically traces all LLM calls made within workflows. LangChain and LlamaIndex, which may be used for custom RAG pipelines, also have first-class Langfuse callback handlers. The Graphiti entity extraction pipeline, which makes multiple LLM calls per episode, can be instrumented to send traces to Langfuse. The result is a unified observability view across all LLM usage in the system, regardless of which component initiated the call.

The analytics that Langfuse provides are critical for the team's operational awareness. Cost dashboards show spending broken down by model, workflow, agent, and workspace. Latency dashboards reveal which workflows are slow and which LLM calls are the bottleneck. Quality dashboards allow the team to score agent outputs and track quality trends over time. Error dashboards surface failed calls, rate limit hits, and provider outages. For a team running AI agents as employees, this observability is not a luxury — it is the equivalent of the performance reviews, expense reports, and incident reports that any manager needs to oversee their workforce.

Langfuse also provides prompt management and experimentation capabilities. The team can version prompts, run A/B tests between different prompt variants, and measure the impact on output quality and cost. This is particularly valuable when optimizing agent workflows, because small changes in prompting can have large effects on both quality and token consumption.

Self-hosted Langfuse runs as a Docker container with a PostgreSQL backend, fitting naturally into the stack's PostgreSQL-centric infrastructure. All trace data, analytics, and prompt versions are stored in the team's own database, ensuring that sensitive information about agent behavior and workspace content never leaves the team's infrastructure.

## Chapter 11: MinIO as the Artifact Storage Engine

The collaboration system generates a continuous stream of artifacts — images produced by creative agents, documents drafted by writing agents, code patches proposed by development agents, analysis reports generated by research agents, architecture diagrams created by planning agents. These artifacts need reliable, scalable, and accessible storage with fine-grained access control.

MinIO provides S3-compatible object storage that can be self-hosted on the team's infrastructure. Its S3 compatibility is native rather than emulated — AWS SDKs, CLI tools, and any S3-compatible client work without modification, with consistent sub-ten-millisecond latency and a target of eleven nines (99.999999999 percent) data durability. Agent workflows that generate files store them through the standard S3 API. Panel frontends that display artifacts retrieve them through pre-signed URLs with time-limited access. Knowledge graph pipelines that index artifact content fetch them through the same S3 interface.

MinIO's architecture is built for reliability through Reed-Solomon erasure coding, a data protection technique that shards each object into data blocks and parity blocks distributed across drives. The default configuration uses N/2 data shards and N/2 parity shards — for a twelve-drive setup, six data and six parity — which means the system can lose up to half of its drives and still recover all data. Each erasure set contains two to sixteen drives (configurable up to thirty-two in recent releases), and every read and write operation includes bitrot protection through checksums that detect and correct silent data corruption. The storage efficiency is dramatically better than traditional replication: for ten petabytes of data, erasure coding requires fifteen to twenty petabytes of raw storage compared to thirty or more petabytes with three-way replication. For production distributed deployments, MinIO requires a minimum of two servers with four drives each, and a sixteen-server deployment with two hundred drives per node can continue serving even if four entire servers go offline simultaneously. All nodes should be homogeneous and use XFS, ZFS, or Btrfs filesystems for strict read-after-write and list-after-write consistency. For the collaboration system's scale, a single MinIO server with local storage provides sufficient capacity, but the architecture supports seamless horizontal scaling when storage needs grow.

Supabase Storage integrates with MinIO as its backing store, which means the access control policies defined in Supabase's Row Level Security system extend to artifact storage. An agent working in workspace A cannot access artifacts from workspace B. A human reviewer can only view artifacts from workspaces where they have membership. The storage layer inherits the same security model as the rest of the data platform.

## Chapter 12: n8n as the Integration and Automation Backbone

Not every workflow in the collaboration system involves AI agents. Many operational processes — syncing data between services, responding to GitLab webhooks, sending notifications, generating periodic reports, cleaning up stale resources — are deterministic automation tasks that don't require LLM calls. n8n provides the workflow automation platform for these non-AI processes.

n8n offers a visual workflow builder with over four hundred pre-built integrations covering everything from HTTP webhooks to database operations to messaging platforms to cloud services. When a merge request is opened in GitLab, an n8n workflow can automatically create a corresponding task object in the studio workspace, notify relevant team members, and trigger a code review agent workflow in Dify. When an agent workflow completes in Dify, an n8n workflow can post the results to the team's messaging channel, update a tracking spreadsheet, and archive the generated artifacts. When a new team member is provisioned in Authentik, an n8n workflow can set up their default workspace memberships, notification preferences, and agent delegation rules.

The distinction between Dify and n8n is important. Dify handles AI-native workflows where LLM calls, knowledge retrieval, and human-in-the-loop decisions are central. n8n handles integration workflows where the goal is to connect systems, transform data, and automate procedures. Together, they cover the full spectrum of automation needs without forcing AI capabilities into workflows that don't need them or forcing deterministic logic into an AI orchestration framework.

n8n is self-hosted and stores its workflow definitions and execution logs in PostgreSQL, continuing the stack's pattern of consolidating state in a single database technology. Its webhook triggers and HTTP nodes provide the glue that connects the various components of the stack, enabling event-driven automation across the entire collaboration system.

## Chapter 13: How the Components Compose into a Working System

The individual components described in the previous chapters are powerful in isolation, but their true value emerges from how they compose into a coherent system. This chapter traces the lifecycle of a typical collaboration scenario to illustrate how data and control flow through the stack.

Consider a scenario where a human team member opens the studio and creates a new architecture decision in the flowchart panel. The panel's frontend sends a request through the Supabase-powered API, which writes the new object to PostgreSQL. Row Level Security verifies that the human has write access to this workspace. Supabase Realtime detects the write through PostgreSQL's Write-Ahead Log and broadcasts the change to all connected participants. The studio's object repository registers the new entity and assigns it a stable identifier. An n8n workflow, triggered by the database change event, submits the new information as an episode to Graphiti, which extracts entities and relationships and integrates them into the temporal knowledge graph stored in FalkorDB.

Meanwhile, a planning agent that is subscribed to workspace events through Supabase Realtime notices the new architecture decision. The agent's workflow, defined in Dify, begins executing. Its first step queries the temporal knowledge graph through Graphiti to retrieve all existing architecture decisions, their relationships, and any relevant historical context. Graphiti's hybrid retrieval combines semantic similarity search with graph traversal, returning not just the most textually similar decisions but also the decisions that are structurally connected in the knowledge graph. The LLM calls made during this retrieval are traced by Langfuse.

The agent's next step uses the retrieved context to analyze whether the new decision is consistent with existing decisions or introduces conflicts. If conflicts are detected, the workflow reaches a Human Input node in Dify and pauses, presenting the conflict analysis to the human team member for review. The workflow state is persisted in PostgreSQL through Hatchet's durable task execution, so it survives indefinitely until the human responds. When the human reviews the analysis and provides a decision, the workflow resumes.

Based on the human's input, the agent generates an updated architecture document and a set of suggested modifications to related objects. These artifacts are stored in MinIO through Supabase Storage. The agent then submits the updates through the studio's API, which creates new objects and links in the workspace. The approval service, implemented as a Dify workflow with governance policies, evaluates whether the changes can be applied automatically or require additional review. If approval is granted, the changes are committed. Supabase Realtime broadcasts the updates to all connected participants. Graphiti receives the new episodes and updates the temporal knowledge graph, marking any superseded facts as invalid and creating new entities and relationships.

Throughout this entire lifecycle, Langfuse has been tracing every LLM call, recording token usage, latency, and cost. Authentik has been verifying every actor's identity and permissions. Hatchet has been ensuring that each task step is durably executed and properly retried if transient failures occur. The audit log in PostgreSQL has recorded every action for accountability. The activity stream has generated human-readable events for the workspace timeline.

This composition is not achieved through a monolithic integration layer. Each component communicates through well-defined interfaces — SQL for data access, REST and GraphQL APIs for service communication, WebSockets for real-time updates, Redis PubSub for streaming events, the Cypher query language for graph queries, S3 for artifact storage, and MCP for agent-to-knowledge-graph interaction. The components are loosely coupled, meaning any one of them can be upgraded, scaled, or replaced without disrupting the others.

## Chapter 14: Scaling to Massive Agent Fleets

The collaboration system is designed to support a small human team working alongside a large number of AI agents. As the system matures and the team defines more workflows, the number of concurrent agents can grow from a handful to hundreds or even thousands. This section describes how the stack scales to accommodate this growth, informed by recent research on multi-agent coordination at scale.

A March 2026 paper from Google and MIT identified three dominant effects in multi-agent systems: a tool-coordination trade-off (multi-agent overhead hurts tasks requiring many different tools), capability saturation (adding agents yields diminishing returns past a threshold), and topology-dependent error amplification (certain coordination patterns amplify errors while others contain them). Their predictive framework selected the optimal coordination strategy with eighty-seven percent accuracy, suggesting that the choice of agent coordination topology is not one-size-fits-all but must be matched to the task characteristics. The collaboration system's architecture supports multiple coordination patterns. For tasks requiring reasoning transparency and quality assurance — like code review or architecture analysis — a centralized supervisor pattern is appropriate, with a single orchestrator agent decomposing the task, delegating to specialists, and synthesizing results. For tasks requiring diverse tool usage — like web research combined with database queries and document generation — a more adaptive pattern where agents transfer tasks directly based on expertise is more effective. The Dify workflow builder supports both patterns, and Hatchet's task DAGs can model either topology.

The industry has consolidated around five architectural patterns for multi-agent coordination: centralized supervisors, adaptive decentralized networks, mesh architectures, hybrid hierarchies, and event-driven systems. The event-driven pattern is particularly relevant for the collaboration system because it naturally maps to the Supabase Realtime infrastructure. Agents can broadcast events asynchronously, operate autonomously while integrating into broader workflows, and respond to real-time data streams without rigid point-to-point dependencies. The four event-driven sub-patterns identified by Confluent — orchestrator-worker, blackboard (shared knowledge base where agents post and retrieve information), market-based (agents bid on tasks), and hierarchical coordination — can all be implemented on the stack's existing infrastructure. The temporal knowledge graph in FalkorDB functions as a natural blackboard, the Hatchet queue functions as an orchestrator-worker system, and Supabase Realtime channels provide the event distribution layer that connects them.

The first scaling dimension is concurrent workflow execution. Dify's Celery worker pool scales horizontally by adding more workers, each consuming tasks from the workflow_based_app_execution queue. Redis Cluster with sharded PubSub ensures that the streaming event layer scales with the worker count. The team can run dozens of Celery workers across multiple machines, each handling multiple concurrent workflow executions. The practical limit is determined by LLM provider rate limits and cost rather than infrastructure capacity.

The second scaling dimension is task dispatch throughput. Hatchet's PostgreSQL-based queue has been proven to handle over five thousand tasks per second in production. For self-hosted deployments, the pure PostgreSQL mode supports up to one hundred tasks per second before RabbitMQ should be introduced for inter-service communication. The collaboration system's typical workload — dozens of agents each executing multi-step workflows — generates task volumes well within Hatchet's PostgreSQL-only capacity.

The third scaling dimension is real-time event distribution. Supabase Realtime's Phoenix-based architecture is designed for millions of concurrent WebSocket connections. Each workspace is a separate channel, so the event distribution load is partitioned naturally by workspace. Even with hundreds of agents connected to a workspace, the Realtime server handles the broadcast efficiently through Erlang's lightweight process model.

The fourth scaling dimension is knowledge graph growth. As agents process more episodes and the temporal knowledge graph grows, FalkorDB's GraphBLAS engine maintains consistent sub-ten-millisecond query latency. The graph size is naturally bounded by workspace scope — most queries traverse a local neighborhood of the graph rather than scanning the entire structure. Graphiti's entity resolution and temporal invalidation prevent the graph from growing without bound, because superseded facts are marked as invalid rather than duplicated.

The fifth scaling dimension is storage. MinIO supports horizontal scaling by adding servers to a distributed cluster. Artifacts are automatically distributed across servers with erasure coding for reliability. PostgreSQL can be scaled vertically for a long time — modern hardware supports database instances with terabytes of RAM and hundreds of thousands of transactions per second — and horizontally through read replicas for query-heavy workloads.

The multi-tenant fairness guarantees provided by Hatchet ensure that scaling does not lead to resource starvation. Even when one agent has thousands of pending tasks, other agents continue to receive their fair share of execution resources. Dynamic concurrency keys allow the system to enforce per-agent, per-workspace, and per-workflow-type limits that prevent any single participant from monopolizing the system.

## Chapter 15: Governance, Observability, and Trust

Running AI agents as employees requires the same governance structures that organizations apply to human employees — identity verification, permission boundaries, activity logging, performance monitoring, and accountability mechanisms. The stakes are real: research shows that eighty-nine percent of organizations running AI agents have implemented some form of agent observability, with quality issues emerging as the primary production barrier for thirty-two percent of deployments. Multi-agent systems introduce a particularly dangerous failure mode — cascading hallucination risk — where one agent's hallucinated output feeds corrupted data to downstream agents at machine speed, compounding errors before any human can intervene. The NIST AI Risk Management Framework was updated in 2025 with a specific Agentic AI profile that mandates mapping of all agent tool-access permissions, automatic access cutoff if agents exceed token budgets or attempt unauthorized API calls, and validation layers between agent handoffs. The open-source stack provides these governance capabilities through the composition of its components.

Identity and access governance flows through Authentik. Every actor in the system — human or AI — has a verified identity with specific role assignments and permission scopes. These identities are propagated to Supabase through JWT tokens that PostgreSQL's Row Level Security evaluates on every data operation. An agent cannot access data from a workspace it is not assigned to. A human reviewer cannot approve their own agent's actions. A service account cannot escalate its own permissions. These constraints are enforced at the database level, below any application logic, making them extremely difficult to circumvent.

Activity observability flows through multiple channels. Langfuse provides deep LLM observability — what each agent said, what it cost, and how it performed. The PostgreSQL audit log provides data-level accountability — who changed what, when, and through which operation. Supabase Realtime provides collaboration-level awareness — who is active, what is happening now. The Graphiti temporal knowledge graph provides historical understanding — how the system's state evolved over time and what caused each change. Together, these observability layers give the team comprehensive insight into agent behavior at every level of abstraction.

Cost governance is critical when running many agents. Each LLM call consumes tokens that cost money. Langfuse's cost dashboards show spending by agent, workflow, model, and workspace. The team can set budgets and alerts that trigger when spending exceeds thresholds. Dify's workflow design allows cost optimization through model routing — using cheaper models for simple tasks and reserving expensive models for tasks that require advanced reasoning. Hatchet's concurrency limits prevent runaway agents from generating unlimited costs by capping the number of concurrent executions.

Quality governance ensures that agents are producing valuable output rather than generating noise. Langfuse's scoring system allows the team to rate agent outputs and track quality trends over time. Dify's Human Input nodes create natural checkpoints where human judgment enters the workflow. The approval service, implemented as a Dify workflow with configurable policies, gates high-risk actions — code changes, permission modifications, external communications — behind human review. The temporal knowledge graph provides the context needed for informed review, because a human reviewer can see not just what the agent is proposing but what knowledge led to the proposal.

Failure handling is governed by Hatchet's durable execution model. When an agent task fails — due to an LLM provider outage, a network error, or an unexpected response — Hatchet automatically retries the task according to configurable retry policies. If the failure persists, the task is marked as failed and surfaced through the monitoring system. Dify's workflow error handling allows graceful degradation — a failed step can trigger a fallback path, notify a human, or record the failure for later analysis. The temporal knowledge graph records the failure as an event, so future agents working on related tasks can be aware that a previous attempt failed and why.

The collaboration system should also implement circuit breaker patterns that halt processing when consistency checks fail. When an agent produces output that fails validation — a code patch that doesn't compile, a factual claim that contradicts the knowledge graph, a cost estimate that exceeds budgets — the circuit breaker prevents the output from propagating to downstream agents or being committed to the workspace. The validation layers between agent handoffs that NIST mandates can be implemented as Hatchet task steps that check outputs before passing them forward, Dify workflow branches that route suspicious outputs to human review, and Supabase RLS policies that prevent agents from writing data that violates workspace constraints.

The emerging protocol landscape for agent governance also deserves mention. Google's Agent2Agent (A2A) protocol, released in April 2025 and donated to the Linux Foundation with over one hundred and fifty supporting organizations, standardizes how agents communicate and coordinate using JSON-RPC 2.0 over HTTP. Anthropic's Model Context Protocol (MCP) standardizes how agents connect to external tools and data sources. These two protocols are complementary — MCP manages tool usage while A2A manages team coordination — and both can be integrated into the collaboration system. The Graphiti MCP server already provides knowledge graph access through MCP, and the Dify platform can expose agent capabilities through A2A endpoints. As the agent ecosystem matures, adherence to these open protocols will ensure that the collaboration system can interoperate with agents built on any framework, not just the ones defined internally.

## Chapter 16: Deployment Architecture

The entire stack is deployed through Docker Compose for development and single-machine production, with a clear upgrade path to Kubernetes for larger deployments. The base deployment consists of the following containers:

PostgreSQL serves as the shared database for Supabase, Hatchet, Langfuse, n8n, Authentik, and application data. A single PostgreSQL instance with the pgvector extension installed handles all data needs. For production, the team should configure streaming replication with at least one replica for read scaling and disaster recovery.

Supabase runs as a set of containers — PostgREST, GoTrue, Realtime, Storage, Kong, and the Studio dashboard — all connecting to the shared PostgreSQL instance. The Supabase containers are stateless and can be scaled horizontally behind a load balancer.

Authentik runs as two containers — the core Django server and a background worker — connecting to the shared PostgreSQL instance. Outpost containers for proxy and LDAP authentication are deployed alongside the applications they protect.

FalkorDB runs as a single container exposing the Redis protocol on its default port. It stores the temporal knowledge graph data in a persistent volume. For production, FalkorDB supports clustering across multiple containers for horizontal scaling.

Redis runs as a single container or cluster providing caching, PubSub, and rate limiting services. It is configured with appropriate memory limits and eviction policies to prevent memory exhaustion.

Dify runs as a set of containers — the API server, the web frontend, the Celery worker pool, and a sandbox for code execution. The workers connect to Redis for task queuing and PubSub, and to PostgreSQL for state persistence.

Hatchet runs as two containers — the engine and the API server — connecting to the shared PostgreSQL instance. Application workers connect to the Hatchet engine through gRPC.

Langfuse runs as a single container with a web interface, connecting to the shared PostgreSQL instance for trace and analytics storage.

MinIO runs as a single container or distributed cluster providing S3-compatible object storage with persistent volumes for data durability.

n8n runs as a single container connecting to the shared PostgreSQL instance for workflow and execution storage.

The studio application, console application, and admin application run as statically served frontends behind Nginx or Caddy, with Authentik providing forward authentication and the reverse proxy routing API requests to the appropriate backend services.

This deployment architecture consolidates state management in PostgreSQL, avoids unnecessary service duplication, and keeps operational complexity manageable for a small team. Each component can be independently restarted, upgraded, or scaled without affecting the others. The entire stack can be brought up with a single docker compose up command and torn down just as easily.

## Chapter 17: The Complete Component Map

To summarize the complete stack and the role each component plays:

PostgreSQL with pgvector is the converged data foundation, handling relational data, document storage, full-text search, vector similarity search, and temporal modeling. It is the single source of truth for all persistent state in the system.

Supabase is the application data platform, providing auto-generated APIs, authentication, real-time change broadcasting, and object storage with Row Level Security policies that enforce workspace boundaries and permission scopes at the database level.

Authentik is the identity and access governance layer, managing human users, AI agent identities, service accounts, role assignments, and authentication flows across multiple protocols including OAuth, OIDC, LDAP, and RADIUS.

Dify is the AI agent orchestration engine, providing visual workflow design, multi-step execution with human-in-the-loop capabilities, multi-provider LLM integration, and a plugin architecture for extending agent capabilities.

Hatchet is the durable task execution engine, providing reliable task queuing, dispatch, retry, and coordination with PostgreSQL-backed transactional guarantees, multi-tenant fairness, and sticky worker assignment.

Graphiti is the temporal knowledge graph framework, providing bi-temporal entity and relationship management, LLM-powered entity extraction, hybrid retrieval combining semantic search with graph traversal, and an MCP server interface for agent interaction.

FalkorDB is the graph database engine, providing high-performance graph storage and query execution through GraphBLAS sparse matrix algebra, with built-in vector indexing for hybrid graph-vector queries.

Redis is the real-time communication and caching layer, providing message brokering for workflow streaming events, in-memory caching for frequently accessed data, and atomic operations for rate limiting.

Langfuse is the LLM observability platform, providing trace collection, cost analytics, latency monitoring, quality scoring, and prompt management across all LLM interactions in the system.

MinIO is the artifact storage engine, providing S3-compatible object storage with erasure coding for reliability and integration with Supabase Storage for access control.

n8n is the integration and automation backbone, providing visual workflow design for non-AI processes, with pre-built integrations for connecting the stack's components and external services.

Together, these eleven components form a complete, self-hosted, open-source infrastructure for real-time human-AI collaboration. Each component was chosen because it excels at its specific role, integrates cleanly with the others through standard protocols and interfaces, and can be independently operated, scaled, and upgraded. The result is a system where a small team can work alongside a massive fleet of AI agents with the reliability, governance, and observability that production collaboration demands.
