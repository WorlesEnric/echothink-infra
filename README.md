# EchoThink Infrastructure

Self-hosted, open-source infrastructure stack for real-time human-AI collaboration. Designed for small teams working alongside AI agents with identities, permissions, task assignments, and observable outputs.

## Architecture Overview

EchoThink wires together 14 open-source components into a cohesive platform:

| Layer | Services | Purpose |
|-------|----------|---------|
| **Foundation** | PostgreSQL 16 (pgvector), Redis 7, MinIO, Nginx | Database, caching, object storage, reverse proxy |
| **Identity** | Authentik | SSO/OIDC provider for all services |
| **Data Platform** | Supabase (PostgREST, GoTrue, Realtime, Storage, Kong) | Application backend with real-time subscriptions |
| **LLM Gateway** | LiteLLM | Unified API for OpenAI, Anthropic, Azure with load balancing |
| **Agent Infra** | Dify, Hatchet, Graphiti + FalkorDB | AI workflows, task orchestration, knowledge graph |
| **Observability** | Langfuse | LLM tracing and cost tracking |
| **Automation** | n8n | Workflow automation and webhook routing |
| **Applications** | Outline, GitLab EE | Knowledge wiki, code collaboration |

```
                         Internet
                            │
                      ┌─────┴─────┐
                      │   Nginx   │  TLS termination, reverse proxy
                      │  :80/:443 │
                      └─────┬─────┘
                            │
              ┌─────────────┼─────────────────────────┐
              │             │                         │
         Authentik     Supabase Kong         Application Services
         (fwd auth)    (API gateway)     (Outline, GitLab, Dify, etc.)
              │             │                         │
              └──────┬──────┴─────────────────────────┘
                     │
         ┌───────────┼───────────┬──────────────┐
         │           │           │              │
     PostgreSQL    Redis      MinIO         FalkorDB
     + pgvector   (cache)     (S3)         (graph DB)
```

## Prerequisites

- **Docker Engine** 24.0+ with Docker Compose v2
- **GNU Make**
- **OpenSSL** (for secret generation)
- At least **16 GB RAM** (32 GB recommended for full stack)
- At least **50 GB disk** for volumes

## Quick Start

### 1. Clone and initialize

```bash
git clone <repository-url> echothink-infra
cd echothink-infra
make init
```

`make init` performs the following:
- Copies `.env.example` to `.env`
- Generates random passwords for PostgreSQL and all 8 service databases
- Generates secrets for Authentik, LiteLLM, Langfuse, n8n, Dify, Outline, Hatchet
- Creates Supabase JWT tokens (anon key + service role key) with 5-year expiration
- Generates self-signed SSL certificates for `*.localhost`

### 2. Configure LLM provider keys

Edit `.env` and add your API keys:

```bash
# At minimum, set one of these:
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
```

### 3. Start core infrastructure

```bash
make up        # PostgreSQL, Redis, MinIO, Nginx only
make up-all    # Full stack (all 14 services)
```

### 4. Verify

```bash
make healthcheck   # Check all service health endpoints
make ps            # Show running containers
```

## System Configuration

### Environment Variables

All configuration is managed through `.env` (generated from `.env.example` by `make init`). Variables are organized into sections:

| Section | Key Variables | Notes |
|---------|-------------|-------|
| **General** | `DOMAIN` | Set to your domain (default: `localhost`) |
| **PostgreSQL** | `POSTGRES_PASSWORD`, `*_DB_PASSWORD` (x8) | One master + per-service passwords |
| **Redis** | `REDIS_URL` | Default: `redis://redis:6379` |
| **MinIO** | `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD` | S3-compatible credentials |
| **Supabase** | `SUPABASE_JWT_SECRET`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` | Auto-generated by init |
| **Authentik** | `AUTHENTIK_SECRET_KEY`, `AUTHENTIK_BOOTSTRAP_PASSWORD` | Identity provider secrets |
| **LiteLLM** | `LITELLM_MASTER_KEY`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY` | LLM gateway + provider keys |
| **Langfuse** | `LANGFUSE_SECRET_KEY`, `LANGFUSE_NEXT_AUTH_SECRET`, `LANGFUSE_SALT` | Observability platform |
| **n8n** | `N8N_ENCRYPTION_KEY` | Workflow automation |
| **Dify** | `DIFY_SECRET_KEY` | AI application builder |
| **Outline** | `OUTLINE_SECRET_KEY`, `OUTLINE_UTILS_SECRET` | Knowledge wiki |
| **GitLab** | `GITLAB_ROOT_PASSWORD`, `GITLAB_SHARED_RUNNERS_TOKEN` | Code collaboration |
| **Hatchet** | `HATCHET_CLIENT_TOKEN`, `HATCHET_JWT_SECRET` | Task orchestration |
| **SMTP** | `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD` | Email delivery |
| **OAuth** | `*_OAUTH_CLIENT_ID`, `*_OAUTH_CLIENT_SECRET` (x8) | Populated after Authentik setup |

### Database Setup

PostgreSQL 16 hosts 8 separate databases with dedicated roles (least privilege):

| Database | Role | Service |
|----------|------|---------|
| `supabase` | `supabase_admin` | Supabase (PostgREST, GoTrue, Realtime, Storage) |
| `authentik` | `authentik_user` | Authentik identity provider |
| `dify` | `dify_user` | Dify AI workflows |
| `hatchet` | `hatchet_user` | Hatchet task orchestration |
| `langfuse` | `langfuse_user` | Langfuse observability |
| `n8n` | `n8n_user` | n8n workflow automation |
| `outline` | `outline_user` | Outline wiki |
| `litellm` | `litellm_user` | LiteLLM key management + spend tracking |

All databases have these extensions enabled: `pgvector`, `pg_trgm`, `btree_gist`, `pg_stat_statements`, `uuid-ossp`.

PostgreSQL is tuned for the convergent workload (`services/postgres/conf/postgresql.conf`):
- `max_connections = 300` (supports all services)
- `shared_buffers = 1GB`
- `wal_level = logical` (required by Supabase Realtime)
- `max_replication_slots = 10`

### Network Architecture

Two Docker networks isolate traffic:

| Network | Type | Purpose |
|---------|------|---------|
| `echothink-internal` | Bridge, `internal: true` | Inter-service communication (no internet access) |
| `echothink-public` | Bridge | Services needing outbound internet (LiteLLM, GitLab, etc.) |

Services like PostgreSQL and Redis are on `echothink-internal` only. Services needing external access (LiteLLM for API calls, GitLab for webhooks) are on both networks. Nginx bridges both networks to route external traffic.

### Reverse Proxy (Nginx)

Nginx routes by subdomain. Each service has a config in `services/nginx/conf.d/`:

| Subdomain | Upstream | Port |
|-----------|----------|------|
| `supabase.${DOMAIN}` | supabase-kong | 8000 |
| `auth.${DOMAIN}` | authentik-server | 9000 |
| `dify.${DOMAIN}` | dify-web / dify-api | 3000 / 5001 |
| `litellm.${DOMAIN}` | litellm | 4000 |
| `outline.${DOMAIN}` | outline | 3000 |
| `gitlab.${DOMAIN}` | gitlab | 8929 |
| `langfuse.${DOMAIN}` | langfuse | 3000 |
| `n8n.${DOMAIN}` | n8n | 5678 |
| `minio.${DOMAIN}` | minio | 9000 / 9001 |
| `hatchet.${DOMAIN}` | hatchet-api | 8080 |

WebSocket support is enabled for Supabase Realtime, Dify, and n8n. Each config includes commented-out Authentik forward auth snippets.

For local development, add entries to `/etc/hosts`:
```
127.0.0.1  supabase.localhost auth.localhost dify.localhost litellm.localhost
127.0.0.1  outline.localhost gitlab.localhost langfuse.localhost n8n.localhost
127.0.0.1  minio.localhost hatchet.localhost
```

### SSL/TLS

`make init` generates self-signed wildcard certificates for `*.${DOMAIN}`. For production:

1. Replace certs in `services/nginx/ssl/` with real certificates, or
2. Use Let's Encrypt with certbot, or
3. Deploy behind a load balancer that terminates TLS

### Identity & SSO (Authentik)

Authentik provides centralized authentication for all services via OAuth2/OIDC. Pre-configured blueprints in `services/authentik/blueprints/` set up:

- **Default auth flow**: Username/email + password + optional TOTP
- **OIDC providers**: Pre-configured for Outline, GitLab, Dify (client IDs: `outline`, `gitlab`, `dify`)
- **Groups**: `admins`, `team-members`, `agents`, `service-accounts`
- **Agent service accounts**: Token-based auth flow for AI agents

After first boot, access Authentik at `https://auth.${DOMAIN}` with the bootstrap credentials from `.env`. Then:

1. Complete the initial setup wizard
2. Note the OAuth client secrets for each provider
3. Update the `*_OIDC_CLIENT_SECRET` variables in `.env`
4. Restart the affected services: `make restart`

### LLM Gateway (LiteLLM)

LiteLLM provides an OpenAI-compatible API at `http://litellm:4000/v1` for all downstream services. Model groups defined in `services/litellm/config.yaml`:

| Group | Models | Use Case |
|-------|--------|----------|
| `high-quality` | Claude Opus 4.6, GPT-4o | Complex reasoning tasks |
| `cost-effective` | Claude Haiku 4.5, GPT-4o-mini | Routine tasks, lower cost |
| `fast` | Claude Haiku 4.5, GPT-4o-mini | Latency-sensitive requests |
| `code` | Claude Sonnet 4.6, GPT-4o | Code generation and analysis |
| `embedding` | text-embedding-3-small, text-embedding-3-large | Vector embeddings |

Routing uses latency-based strategy with automatic failover between providers. The admin dashboard is at `https://litellm.${DOMAIN}`.

### Object Storage (MinIO)

MinIO provides S3-compatible storage. Buckets created on initialization:

| Bucket | Used By |
|--------|---------|
| `supabase-storage` | Supabase Storage API |
| `outline-data` | Outline file attachments |
| `dify-storage` | Dify uploaded files |
| `artifacts` | General artifact storage |
| `backups` | Automated backup destination |
| `gitlab-artifacts` | GitLab CI/CD artifacts |
| `gitlab-uploads` | GitLab user uploads |
| `gitlab-lfs` | GitLab LFS objects |

MinIO Console is accessible at `https://minio.${DOMAIN}` (port 9001 in dev mode).

## Operations

### Make Targets

| Command | Description |
|---------|-------------|
| `make init` | First-time setup: generate secrets, SSL certs, `.env` |
| `make up` | Start core infrastructure (PostgreSQL, Redis, MinIO, Nginx) |
| `make up-all` | Start all services |
| `make down` | Stop core infrastructure |
| `make down-all` | Stop all services |
| `make restart [s=SERVICE]` | Restart all or a specific service |
| `make logs [s=SERVICE]` | Tail logs for all or a specific service |
| `make ps` | Show running containers |
| `make shell-db` | Open interactive psql session |
| `make healthcheck` | Run health checks on all services |
| `make backup` | Backup PostgreSQL + MinIO + FalkorDB |
| `make restore f=PATH` | Restore from a backup archive |
| `make pull` | Pull latest container images |
| `make build` | Build custom images (PostgreSQL, LiteLLM) |
| `make clean` | Remove all containers, volumes, and data (destructive) |

### Backup & Restore

Automated backup covers:
- **PostgreSQL**: `pg_dump` of all 8 databases in custom format
- **MinIO**: Full bucket mirror
- **FalkorDB**: RDB snapshot

```bash
make backup                              # Full backup to ./backups/YYYYMMDD_HHMMSS/
bash scripts/backup.sh /path/to/backup   # Custom backup directory
bash scripts/backup.sh --pg-only         # PostgreSQL only
```

See `docs/operations/backup-restore.md` for detailed procedures and disaster recovery runbook.

### Health Checks

```bash
make healthcheck
```

Checks all 14 services with colored output. Verifies:
- PostgreSQL connectivity and all 8 databases exist
- Redis and FalkorDB responding to ping
- MinIO health endpoint and bucket existence
- HTTP health endpoints for all application services
- Outline and GitLab marked as optional (may not be running)

## Kubernetes Deployment

Helm chart and Kustomize overlays are provided in `k8s/` for production Kubernetes deployment.

### Helm

```bash
# Development
helm install echothink k8s/helm/echothink/ -n echothink --create-namespace

# Production
helm install echothink k8s/helm/echothink/ -n echothink --create-namespace \
  -f k8s/helm/echothink/values.production.yaml
```

Production values include:
- Higher resource limits and requests
- Multiple replicas for stateless services
- HPA for LiteLLM, Dify workers, Supabase Realtime
- Pod anti-affinity rules
- NetworkPolicies for service isolation
- Ingress with cert-manager for TLS

### Kustomize

```bash
# Development
kubectl apply -k k8s/kustomize/overlays/dev/

# Production
kubectl apply -k k8s/kustomize/overlays/production/
```

## Adding a New Service

1. Create `services/<service-name>/docker-compose.<service-name>.yml`
2. Add Nginx config to `services/nginx/conf.d/<service-name>.conf`
3. Add Authentik OIDC provider blueprint to `services/authentik/blueprints/`
4. Add database to `services/postgres/init/01-databases.sql` and role to `02-roles.sql`
5. Add environment variables to `.env.example`
6. Add the compose file to `docker-compose.apps.yml` includes
7. Add health check to `scripts/healthcheck.sh`

See `CLAUDE.md` for naming conventions and patterns.
